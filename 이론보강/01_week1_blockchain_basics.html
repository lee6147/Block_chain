Week 1. 블록체인 기초: 공유 장부, 해시, 체인

블록체인은 여러 노드가 같은 원장 사본을 보관하고, 블록의 데이터와 이전 해시를 연결해 변경 시도를 빠르게 드러내는 구조다.

핵심 개념 해시 시뮬레이터 분산 장부 비교 API/체인 데이터 비교 실습 연결

정석 개념 요약

블록

거래나 상태 변경 기록을 일정 단위로 묶은 장부의 한 페이지다. 데이터, 이전 해시, 현재 해시가 함께 의미를 만든다.

해시

입력을 고정 길이 값으로 요약한 결과다. 같은 입력은 같은 해시를 만들지만, 한 글자만 바뀌어도 결과가 크게 바뀐다.

체인

현재 블록이 이전 블록의 해시를 기억하는 연결 구조다. 앞 블록이 바뀌면 뒤 블록의 이전 해시와 맞지 않아 연결이 깨진다.

합의

단순 과반수 손들기가 아니라, 각 노드가 같은 유효성 규칙을 적용하고 네트워크의 합의 방식과 최종성 조건을 통과한 기록을 정식 상태로 받아들이는 과정이다.

개념 보강: 비유 → 정석 정의 → 확인 질문

블록은 장부의 한 페이지, 해시는 데이터 지문, 체인은 이전 페이지 지문을 다음 페이지에 적는 고리다. 합의는 "많이 가진 쪽이 맞다"가 아니라 "정해진 검증 규칙을 통과한 후보 중 네트워크 규칙이 정식 기록으로 확정한 것"을 받아들이는 과정이다. 확인 질문은 간단하다. "데이터 한 글자가 바뀌면 어느 값이 바뀌고, 노드들은 무엇을 기준으로 정답을 고르는가?"

중앙화 시스템은 한 데이터베이스가 원본인 반면, 탈중앙화 블록체인은 여러 참여자가 정해진 규칙으로 올바른 기록을 선택한다. 그래서 블록체인은 단순 저장소가 아니라 기록, 검증 규칙, 네트워크 합의가 결합된 시스템이다.

시뮬레이터 1: 블록 데이터 수정과 해시 변경

왜 해보는가: 블록체인은 “수정이 물리적으로 불가능한 파일”이 아니라, 데이터가 바뀌면 해시와 연결 상태가 즉시 어긋나는 구조라는 점을 눈으로 확인해야 한다.

해시 안전난간: 이 페이지의 hash는 변화 감지를 보여주는 교육용 단순 해시다. 실제 블록체인은 SHA-256, Keccak 계열처럼 충돌 저항성을 갖춘 암호학적 해시를 사용한다.

예측 → 실행 → 상태 변화 → 해석

먼저 Block #2 문장을 바꾸면 hash가 바뀔 것이라고 예측한다. 실행 후 Block #3의 previousHash 불일치를 보고, 체인 다시 계산은 "로컬 구조를 다시 맞추는 과정"으로 해석한다. 실제 네트워크에서는 이것만으로 끝나지 않고 합의 규칙을 통과해야 정식 기록이 된다.

실제 실습 연결: Week1 가격 조회 앱은 API 데이터를 읽고 화면에 표시했다. 블록체인 앱은 가격 대신 블록 번호, 거래 해시, 이전 해시 같은 검증 가능한 데이터를 읽고 표시한다.

시뮬레이터 2: 중앙 장부와 분산 장부 비교

왜 해보는가: 한 서버만 원본이면 서버 기록이 바뀔 때 사용자는 그대로 받아들인다. 분산 장부에서는 각 노드가 자신이 받은 기록을 같은 규칙으로 검증하고, 임의로 바뀐 기록은 정식 상태와 어긋난 후보로 드러난다.

예측 → 실행 → 상태 변화 → 해석

중앙 서버 수정은 사용자 모두의 화면이 바뀔 것으로 예측한다. 분산 장부에서 노드 하나만 임의 수정하면 "4표 대 1표"가 아니라 "같은 검증 규칙 결과와 불일치"가 표시된다. 실제 합의는 stake, hash power, validator set, finality 같은 네트워크별 규칙을 함께 본다.

중앙 장부

서버 원본: Block A: 정상 출석부

사용자 1사용자 2사용자 3

분산 장부

5개 노드가 같은 기록과 같은 검증 결과를 보관 중

합의 오해 방지

이 화면은 불일치를 발견하는 입문 모델이다. 실제 블록체인은 노드 개수만 세지 않고 블록 형식, 이전 해시, 서명, 잔액, 수수료, 난이도나 지분, 최종성 규칙을 종합해 정식 기록을 판단한다.

실제 실습 연결: CoinGecko API 오류는 외부 API 서비스 문제이고, 블록체인 기록 불일치는 원장 검증 문제다. 둘을 구분해야 Week2 이후 RPC, explorer, 거래 실패를 정확히 해석할 수 있다.

오해 방지: 여러 노드가 같은 값을 보인다는 표시는 "불일치가 적다"는 관찰일 뿐이다. 실제 네트워크에서는 단순 노드 수가 아니라 합의 규칙과 최종성 조건을 통과해야 정식 기록으로 인정된다.

시뮬레이터 3: API 데이터와 블록체인 데이터 읽기 비교

왜 해보는가: Week1 가격 앱과 블록체인 앱은 둘 다 데이터를 읽어 화면에 표시한다. 하지만 가격 API는 서비스 제공자의 응답이고, 블록체인 데이터는 네트워크가 합의한 원장 상태를 RPC나 Explorer로 읽는다는 차이가 있다.

예측 → 실행 → 상태 변화 → 해석

가격 데이터 버튼은 표가 바뀔 것이라고 예측한다. 블록 데이터 버튼은 blockNumber, tx count, previousHash가 카드로 보일 것이라고 예측한다. 429 제한은 블록체인 실패가 아니라 API 서비스 제한으로 해석한다. 이 시뮬레이터는 실제 지갑, RPC, 체인을 호출하지 않는 모의 데이터다.

1. 먼저 예측

가격 API는 누가 값을 제공하는지, 블록 데이터는 어떤 필드로 검증 단서를 주는지 말해 본다.

2. 버튼 실행

왼쪽은 서비스 응답과 제한을 보고, 오른쪽은 블록 번호와 이전 해시 연결을 본다.

3. 해석 분리

API 오류, RPC 조회 실패, 체인 상태 변화, 거래 실패를 같은 문제로 묶지 않는다.

대기 중: 먼저 "가격 데이터 읽기"를 누르고, 값의 출처가 API 서비스라는 점을 확인하세요.

RPC/Explorer 흐름

RPC블록 JSON검증 카드
대기 중: "블록 데이터 읽기"를 누르고, previousHash가 무엇을 검증하게 해 주는지 확인하세요.

실제 실습에서 어디에 해당하는가: Week1의 fetch(), JSON 파싱, 429 오류 처리는 Week2 이후 RPC 조회와 Explorer 확인의 웹 기초다. 단, 데이터의 신뢰 원천은 서비스 API와 블록체인 원장으로 다르다.

RPC/Explorer 안전난간: RPC와 Explorer는 체인 데이터를 읽는 창구다. 제공자 화면을 믿는 것과 직접 노드로 검증하는 것은 구분해야 하며, 중요한 판단에서는 네트워크, block number, finality 상태를 함께 확인한다.
