# Structure And UX Review

## Verdict

**CONDITIONAL**

페이지별 기본 구조는 충분하지만, 전체 코스 구조와 반복 학습 패턴에서 수업 전 수정할 지점이 있다.

## 잘된 점

- 대부분 페이지가 `핵심 개념 -> 시뮬레이터 -> 실습 연결` 순서를 유지한다.
- 모바일 overflow evidence는 0으로 확인된다.
- 각 HTML은 reset 동작과 여러 인터랙션을 제공한다.
- Week1, Week2, Week5, Week6은 “오류를 도구 고장으로 묶지 않고 원인을 좁힌다”는 교육 목표와 잘 맞는다.

## 구조 리스크

## 주의 사항

- 구조 수정은 Week3 프레이밍과 전체 주차 이동 nav부터 처리한다.
- 모바일 표/컨트롤 개선은 P2로 두고, P1 수정 뒤 검증이 깨지지 않는지 먼저 확인한다.
- Ralph 수정 루프에 들어가면 한 번에 대형 리디자인을 하지 않고 `P1 수정 -> html 검증 -> evidence 확인 -> 재심사` 순서로 진행한다.

### 1. Week3 설명 패턴 불일치

Week3은 내용 자체는 강하지만, 다른 주차에 비해 시뮬레이터 앞의 학습 목적 문장이 덜 통일되어 있다. 자동 계약은 통과하지만 초보자가 “왜 이걸 눌러야 하는지”를 읽는 흐름에서는 약점이다.

권고 형식:

```text
왜 해보는가:
예측 -> 실행 -> 상태 변화 -> 해석:
실제 실습에서 어디에 해당하는가:
```

### 2. 전체 커리큘럼 내비게이션 부족

각 주차 HTML을 직접 열면 이전/다음 주차 이동이 분명하지 않다. 초보자 교육 자료는 단일 페이지 품질뿐 아니라 “다음에 뭘 봐야 하는가”가 중요하다.

권고:

```html
<a href="index.html">목차</a>
<a href="02_week2_wallet_transaction.html">이전</a>
<a href="04_week4_stablecoin_networks.html">다음</a>
```

### 3. 모바일 정보량

overflow는 없지만, Week4~6은 표와 컨트롤 정보량이 많다. 375px 화면에서 깨지지는 않아도 학습 부담은 남는다.

권고:

- 긴 표 앞에 `핵심만 보기` 카드 추가
- 오류 진단은 먼저 3~5개 대분류로 묶고 세부 메시지를 뒤에 둔다.
- Week6은 추천 첫 경로를 고정한다.

### 4. Index 필터 관점

현재 index의 필터는 제작 이력 기준에 가깝다. 학생에게는 “왜 이 자료가 새로 만들어졌는가”보다 “내가 지금 어디로 가야 하는가”가 더 중요하다.

권고 필터:

- `처음부터`
- `지갑/트랜잭션`
- `오류 해결`
- `ERC20/Staking`

## 수업 전 최소 수정안

1. Week3의 두 시뮬레이터 앞뒤 설명을 다른 주차와 같은 형식으로 맞춘다.
2. 모든 Week 페이지에 `목차 / 이전 / 다음` 링크를 추가한다.
3. Week6 첫 화면에 추천 진행 순서를 추가한다.

이 세 가지만 반영해도 최종 판정은 PASS로 올라갈 가능성이 높다.
