ERC20, Staking, Error Diagnosis

ERC20 staking은 `approve -> stake -> reward 증가 -> claim/withdraw` 순서로 진행된다. 이 페이지는 권한, 잔액, 보상 금고, lock/time, gas, nonce, require 조건을 분리해 보는 오프라인 정적 시뮬레이터이며 실제 지갑이나 RPC를 호출하지 않는다.

목차 이전 Week5
처음 순서 Approve/Stake Reward 오류 판별 정석 개념 학습 목표 실습 연결

오늘 이해할 3문장

1. approve는 송금이 아니다.
토큰은 그대로 있고 allowance만 기록된다.
2. stake가 실제 이동이다.
Staking 컨트랙트가 transferFrom으로 allowance와 balance를 줄인다.
3. 보상은 조회와 수령이 다르다.
pendingReward는 계산이고, claim/withdraw는 상태를 바꾸는 쓰기 트랜잭션이다.
처음이면 이 순서로

approve -> allowance 조회 -> stake -> pendingReward -> claim -> withdraw 순서로 누른다. 초기화는 영역별로 나뉜다. Approve/Stake 초기화는 balance, allowance, staked amount를 되돌리고, Reward 초기화는 보상 흐름을 되돌리며, 오류 판별기 초기 상태는 선택값을 학습 시작 상태로 되돌린다.

비유와 정석 개념 먼저 보기

비유로 먼저 이해하기

approve는 자동이체 한도 등록이고, stake는 등록된 한도 안에서 실제로 돈을 옮기는 단계다. 보상은 사물함에 토큰을 맡긴 시간에 따라 쌓이는 포인트처럼 이해한다.

approve / allowance
spender가 내 토큰을 사용할 수 있는 최대 한도를 기록한다. 이때 사용자 balance는 줄지 않는다.
stake / transferFrom
Staking 컨트랙트가 allowance 범위 안에서 토큰을 실제로 이동한다. 이때 balance, allowance, staked amount가 바뀐다.
reward / claim / withdraw
pendingReward는 계산이고, claim과 withdraw는 reward pool, lock, staked amount 조건을 확인하는 쓰기 트랜잭션이다.
확인 질문

지금 실패가 토큰 balance, allowance, spender 주소, reward pool, lock, gas, nonce 중 어느 문에서 막혔는가? 이 질문에 답한 뒤 아래 버튼을 누른다.

처음 눌러볼 순서

이 페이지는 설명을 오래 읽기보다 아래 6단계를 한 번 눌러 보면서 상태 변화를 보는 구조다.

1approve()송금 아님, allowance 기록
2allowance 조회spender가 Staking인지 확인
3stake()transferFrom 실제 이동
4pendingReward()보상 계산만 확인
5claimReward()보상만 수령
6withdraw()원금 회수
정상 작동 기준

approve 후 balance는 그대로, stake 후 balance와 allowance는 감소, claim 후 staked amount는 유지, withdraw 후 staked amount는 0으로 돌아가야 한다.

이미 50 HNL을 맡긴 상태에서 30 HNL을 추가 stake한다면, 먼저 50 HNL이 지난 시간 동안 만든 reward를 정산하고 그다음 staked amount를 80 HNL로 바꿔야 한다.

Reward 증가 시뮬레이터

왜 해보는가: 보상은 매초 모든 사용자에게 실제로 쓰는 것이 아니라, `stake`, `claimReward`, `unstake` 같은 상태 변경 함수가 호출될 때 lazy update로 계산된다. reward pool이 비어 있으면 claim도 실패할 수 있다.

단위 안전난간: 이 계산기는 화면 표시 단위(HNL)로 움직이는 교육용 근사 모델이다. 실제 컨트랙트는 `uint256` raw unit을 쓰므로 50 HNL은 decimals=18 기준 `50 * 10^18`로 변환한 뒤 계산하고, 화면에는 다시 formatUnits로 보여준다.

claim vs withdraw

claimReward는 보상만 이동시키고 staked amount를 유지한다. 이 페이지의 teaching mode는 claimReward를 먼저 눌러 보상과 원금 회수를 분리한다. 실제 컨트랙트에서는 withdraw가 보상을 자동 정산하거나 원금만 회수할 수 있으므로 구현별 코드를 확인한다.

정석 개념 설명

ERC20은 이더리움 계열에서 fungible token을 다루기 위한 표준 인터페이스다. `approve(spender, amount)`는 송금이 아니라 `spender`가 내 토큰을 사용할 수 있는 한도를 storage에 기록하는 함수다. 실전에서는 spender 주소와 승인 수량을 확인하고, 무제한 승인과 오래 남은 승인은 revoke/approve(0)로 줄일 수 있어야 한다.

Staking 컨트랙트가 사용자의 토큰을 가져가려면 사용자가 먼저 `token.approve(stakingAddress, amount)`를 호출해야 한다. 실제 이동은 `stake(amount)` 내부에서 `token.transferFrom(msg.sender, address(this), amount)`가 실행될 때 일어난다.

비유로 먼저 이해하기

`approve`는 자동이체 한도 등록이고, `stake`는 등록된 한도 안에서 실제로 돈을 옮기는 단계다. 보상은 사물함에 토큰을 맡긴 시간에 따라 쌓이는 포인트처럼 이해할 수 있다.

개념 보강: 비유 → 정석 정의 → 확인 질문

approve는 허가, allowance는 남은 한도, transferFrom은 실제 이동, reward는 호출 시점에 계산되는 보상이다. 확인 질문은 "지금 실패가 토큰 balance, allowance, spender 주소, reward pool, lock, gas, nonce 중 어느 문에서 막혔는가?"이다.
