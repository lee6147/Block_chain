Week 2. 지갑과 트랜잭션: 서명, RPC, 블록 포함, Explorer 확인

지갑은 코인을 담는 통이 아니라 개인키로 서명하는 도구다. 트랜잭션은 작성, 서명, RPC 전송, 블록 포함, explorer 확인까지 이어져야 성공 여부를 판단할 수 있다.

핵심 개념 트랜잭션 흐름 지갑/RPC 역할 오류 분리

정석 개념 요약

EOA와 주소

EOA는 개인키가 제어하는 일반 지갑 계정이고, 주소는 공개 가능한 식별자다. 주소를 알아도 자산을 움직일 수는 없다.

개인키와 서명

개인키는 트랜잭션에 서명하는 비밀 값이다. 네트워크에는 개인키가 아니라 트랜잭션 데이터와 서명값이 전송된다.

RPC와 Explorer

RPC는 노드에 요청을 보내는 창구이고, Explorer는 블록체인에 기록된 거래 해시, 블록 번호, 상태, gas를 확인하는 조회 도구다.

개념 보강: 비유 → 정석 정의 → 확인 질문

지갑은 돈통이 아니라 도장 보관함, RPC는 접수 창구, Explorer는 공개 조회 화면이다. 확인 질문은 "지갑이 만든 서명과 RPC가 전달한 요청, Explorer가 보여주는 receipt는 각각 어느 단계인가?"이다.

구분맡는 일절대 섞으면 안 되는 지점
Provider체인 상태를 읽거나 RPC 요청을 보낼 연결 객체Provider만 있다고 사용자의 자산을 움직일 수는 없다.
Signer / Wallet개인키 또는 지갑 승인으로 트랜잭션에 서명하는 권한Signer는 권한이다. 주소 조회용 Provider와 같은 말이 아니다.
RPC서명된 트랜잭션을 노드에 접수하거나 블록/잔액을 조회하는 통신 창구RPC는 지갑이 아니며 개인키를 보관하지 않는다.
Receipt블록 포함 뒤 실행 결과, status, gasUsed, blockNumber를 담은 기록tx hash가 생긴 것과 receipt status가 성공인 것은 다르다.
이 페이지의 모든 해시, 블록 번호, receipt는 수업용 가짜 값이다. 실제 MetaMask, RPC, 체인 호출을 실행하지 않는다.

시뮬레이터 1: 트랜잭션 라이프사이클

왜 해보는가: 해시가 생겼다는 사실만으로 성공을 단정하면 안 된다. 서명된 요청이 RPC를 지나 멤풀에 들어가고, 블록에 포함된 뒤 receipt status까지 확인해야 한다.

예측 → 실행 → 상태 변화 → 해석

버튼을 누르기 전 다음 단계가 무엇인지 먼저 말한다. 이 시뮬레이터는 앞 단계가 없으면 뒤 단계가 막히도록 되어 있다. 실행 후 stage 카드와 로그가 바뀌면, 그 상태가 "작성", "서명", "제출", "포함", "receipt 확인" 중 어디인지 해석한다.

시뮬레이터 2: 지갑, Provider, RPC 역할 구분

왜 해보는가: 읽기 연결과 서명 권한을 섞으면 개인키를 잘못 다루거나 RPC를 지갑으로 오해하게 된다. 각 버튼은 같은 앱 안에서도 전혀 다른 경로를 강조한다.

예측 → 실행 → 상태 변화 → 해석

잔액 조회는 서명이 필요 없고, MetaMask 연결은 사용자 승인과 Signer가 필요하다고 예측한다. 강조된 경로가 예상과 다르면 Provider와 Signer 역할을 다시 분리한다.

학생이 자주 하는 말교정 문장다음 확인 위치
"RPC에 지갑을 연결했다"RPC는 노드 통신 창구이고, 지갑 연결은 Signer 권한 요청이다.코드에서 `Provider`와 `Signer` 변수가 분리되어 있는지 본다.
"Provider가 있으니 송금된다"Provider는 읽기/전송 통로이고, 송금 승인에는 Signer 서명이 필요하다.MetaMask 승인 팝업 또는 서버 Wallet 서명 위치를 본다.
"해시가 나왔으니 성공했다"해시는 식별자다. 성공은 receipt의 `status: 1`과 blockNumber까지 본다.Explorer의 Status, Block, Gas Used 영역을 본다.

실제 실습 연결: `BrowserProvider`는 브라우저의 MetaMask 연결을 감싸고, `JsonRpcProvider`는 서버가 Infura 같은 RPC로 읽는 통로다. `Wallet` 또는 `Signer`는 서명 권한이 있는 객체다.

Faucet 문제와 트랜잭션 문제 분리

왜 해보는가: 오류를 “도구 고장”으로 묶지 않고 지갑, 잔고, 네트워크, RPC, 블록 포함, explorer 중 어느 단계 문제인지 좁히기 위해서다.

예측 → 실행 → 상태 변화 → 해석

체크박스를 바꾸기 전 "다음에 볼 화면"을 예측한다. 결과 문장은 Faucet 준비 문제, 네트워크 문제, 제출 전 실패, receipt 확인 중 하나로 원인을 좁히는 해석 문장이다.

초기 상태: Explorer receipt status를 확인하면 된다.

증상먼저 볼 곳receipt까지 이어지는 판단
Faucet에서 ETH가 안 옴Faucet 페이지 메시지, 주소, 네트워크, rate limit아직 트랜잭션 문제가 아니다. 잔액 준비 단계다.
서명 전에 막힘MetaMask 네트워크, 주소 형식, 사용자 승인 여부receipt가 생기기 전이므로 Explorer가 아니라 지갑 화면을 본다.
RPC 제출 후 pendingnonce, 같은 지갑의 pending, RPC 응답, gas 설정tx hash는 있을 수 있지만 아직 blockNumber가 없을 수 있다.
Explorer에 없음tx hash 생성 여부와 Mainnet/Sepolia explorer 선택네트워크가 맞아야 receipt 조회가 가능하다.
블록 포함 후 실패receipt status, gasUsed, revert 또는 잔액/nonce 원인블록 포함은 성공했지만 실행 결과는 실패일 수 있다.

실제 실습 연결: Faucet 실패는 테스트 자금 준비 문제이고, 전송 실패는 서명된 요청의 검증/실행 문제다. 둘을 분리하면 Week3의 gas, nonce, pending 진단으로 자연스럽게 넘어간다.
