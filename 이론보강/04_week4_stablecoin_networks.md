# Week4 스테이블코인 전용 네트워크

## 학습 목표

- 스테이블코인을 "가격 안정 자산"으로 이해하고, 스테이블코인 전용 네트워크를 "결제 UX를 바꾸는 인프라"로 구분한다.
- 일반 네트워크에서 스테이블코인을 쓰는 방식과 스테이블코인 전용 네트워크의 차이를 설명한다.
- gas token이 ETH 같은 변동성 자산일 때와 BaseCoin/USDC/USDT 같은 안정 자산일 때의 사용자 경험 차이를 비교한다.
- L1, L2, rollup, settlement를 수수료 예측성, 최종성, 보안 의존성 관점에서 연결한다.
- 기존 Week4 자료의 StableNet, Plasma, Stable, Arc, Tempo, Maroo 비교를 초심자용 흐름으로 재구성한다.

## 비유로 먼저 이해하기

일반 네트워크에서 스테이블코인을 보내는 것은 "달러 상품권을 택배로 보내는데 배송비는 매일 가격이 바뀌는 기름으로 내야 하는 상황"과 비슷하다. 내가 보내고 싶은 것은 10달러짜리 상품권인데, 배송비를 내기 위해 따로 ETH를 사야 하고 ETH 가격과 네트워크 혼잡도에 따라 배송비가 계속 달라진다.

스테이블코인 전용 네트워크는 "상품권 결제 전용 고속도로"에 가깝다. 결제 차량만 다니고, 통행료도 상품권 또는 원화/달러에 가까운 안정 자산으로 낸다. 사용자는 "이번 송금은 약 10원"처럼 비용을 이해하기 쉽다.

| 비유 | 일반 네트워크 | 스테이블코인 전용 네트워크 |
| --- | --- | --- |
| 도로 | 트럭, 버스, 승용차, 결제 차량이 함께 달리는 일반 고속도로 | 결제/송금 차량 중심의 전용 도로 |
| 보내는 돈 | USDT, USDC 같은 스테이블코인 | 스테이블코인 |
| 통행료 | ETH, WEMIX 같은 변동성 gas token | USDT, USDC, BaseCoin, OKRW 같은 안정 gas token 또는 후원 수수료 |
| 사용자 느낌 | "돈은 달러인데 수수료는 ETH로 준비해야 하네" | "보내는 돈과 수수료 단위가 비슷해서 예측 가능하네" |
| 운영 목표 | 모든 dApp을 처리하는 범용 플랫폼 | 결제, 송금, 정산, 기업 도입에 최적화 |

이 비유에서 핵심은 스테이블코인 자체와 네트워크 UX가 다르다는 점이다. 스테이블코인은 가격을 안정시키려는 자산이고, 스테이블코인 전용 네트워크는 그 자산을 더 쉽게 보내고 정산하도록 만든 실행 환경이다.

## 정석 개념 설명

스테이블코인(stablecoin)은 달러, 원화, 유로 같은 기준 자산에 가치가 연동되도록 설계된 암호화폐다. 정의의 중심은 가격 안정성이다. 예를 들어 USD-pegged 스테이블코인은 1토큰이 1달러에 가깝도록 설계되고, KRW-pegged 스테이블코인은 1토큰이 1원 또는 정해진 원화 가치에 가깝도록 설계된다.

중요한 점은 "가치가 안정되도록 설계되었다"와 "항상 보장된다"가 같지 않다는 것이다. 스테이블코인은 담보, 발행자, 상환 구조, 시장 유동성, 규제, 동결/검열 가능성, 브릿지 보안에 따라 리스크가 달라진다. 전용 네트워크가 수수료 UX를 단순하게 만들 수 있어도, 자산 리스크와 정산 리스크까지 사라지는 것은 아니다.

스테이블코인 전용 네트워크는 가격 안정 자산을 더 잘 전송하기 위한 블록체인 인프라다. 여기서 핵심은 "스테이블코인이 있다"가 아니라 "네트워크 수수료, 최종성, 결제 흐름, 규제/기업 요구, 브릿지 구조가 스테이블코인 사용에 맞춰져 있다"는 점이다.

일반 네트워크에서 USDC나 USDT를 보낼 때도 토큰 자체는 안정적일 수 있다. 하지만 gas token은 ETH처럼 변동성이 큰 자산일 수 있다. 그러면 사용자는 다음 문제를 만난다.

```text
보내는 자산: USDC 10달러
필요한 수수료 자산: ETH
문제: USDC는 있는데 ETH가 없으면 보낼 수 없음
문제: ETH 가격과 네트워크 혼잡도 때문에 수수료 예측이 어려움
```

전용 네트워크는 이 불편을 줄이려 한다. 기존 Week4 자료 기준으로 Plasma는 USDT 결제와 제로 수수료 모델, Stable은 USDT gas 모델, Arc는 USDC gas와 FX 엔진, Tempo는 여러 스테이블코인을 수수료에 쓰는 유연성, Maroo는 KRW-pegged OKRW와 규제 준수, StableNet은 KRW-pegged BaseCoin을 gas token으로 쓰는 결제/송금 중심 구조로 설명된다. 이 상태와 조건은 자료 작성 시점의 수업 자료 기준이며, 최신 운영 상태로 단정하지 않는다.

L1과 L2의 차이도 함께 봐야 한다.

| 개념 | 정석 설명 | 스테이블코인 네트워크에서 보는 이유 |
| --- | --- | --- |
| L1 | 자체 합의와 보안 구조를 가진 기본 블록체인 | 결제 전용 독립 생태계를 만들 수 있음 |
| L2 | L1 위에 구축된 확장 네트워크 | L1 보안을 활용하면서 더 싸고 빠른 실행을 노림 |
| Rollup | 여러 트랜잭션을 L2에서 처리하고 결과를 L1에 기록하는 확장 방식 | 수수료 절감과 처리량 향상 |
| Settlement | 최종 정산이 어디에서 확정되는지의 문제 | 기업 결제에서는 최종성, 출금 시간, 리스크가 중요 |
| Gas token | 수수료를 지불하는 토큰 | 변동성 자산이면 비용 예측이 어렵고, 안정 자산이면 UX가 단순해짐 |

따라서 Week4의 정석 구조는 다음 공식으로 요약할 수 있다.

```text
스테이블코인 = 가격 안정 자산
스테이블코인 전용 네트워크 = 안정 자산을 보내기 위한 비용/속도/정산/UX 최적화 인프라
좋은 결제 UX = 안정적인 송금 금액 + 예측 가능한 수수료 + 빠른 확인 + 쉬운 잔고 관리
```

| 스테이블코인 리스크 | 상품권 발행사와 환불 조건 | 페깅, 담보, 상환, 동결, 브릿지, 정산 구조에서 생기는 위험 | 가격 안정 UX와 실제 안전성을 구분하게 한다 | depeg 사례 토론, bridge/settlement 비교, 규제 준수 항목 | 스테이블코인은 항상 1달러로 안전하다 | 가격, 발행자, 브릿지, 정산 중 어디의 리스크인가? |

네트워크 이름을 외우기 전에 준비물 판단 흐름을 먼저 본다. 1단계는 `보내는 자산`이 무엇인지 확인하는 것이다. 2단계는 수수료를 낼 `gas token`이 무엇인지 확인하는 것이다. 3단계는 최종 정산이 자체 L1에서 끝나는지, L2/rollup을 통해 다른 L1에 기록되는지 확인하는 것이다. 이 세 질문에 답하면 네트워크 이름이 달라도 사용자가 준비해야 할 잔액과 감수해야 할 리스크를 설명할 수 있다.

## 수업 실습과 연결

Week4 기존 자료는 `week/week4/stablecoin_networks.html`, `week/week4/stablenet_explorer.html`, `week/week4/week4_study_guide.md`, `week/week4/Blockchain Lab #4.pdf`로 이어진다.

수업 흐름은 다음처럼 연결하면 초심자가 따라오기 쉽다.

1. 먼저 Week3의 테스트넷 경험을 떠올린다. 학생은 이미 Faucet, 잔고, gas, Explorer, tx hash를 봤다.
2. "테스트넷 ETH가 없으면 아무것도 못 했다"는 경험을 일반 네트워크 UX 문제로 연결한다.
3. USDT/USDC 같은 스테이블코인은 가격이 안정적이지만, 일반 네트워크에서는 gas token이 별도일 수 있음을 설명한다.
4. `stablecoin_networks.html`의 일반 네트워크 vs 전용 네트워크 시뮬레이션을 축약해 보여준다.
5. StableNet 사례에서 BaseCoin이 KRW-pegged gas token으로 소개된 이유를 묻는다.
6. `stablenet_explorer.html`의 Explorer/Faucet 탐색은 "네트워크를 실제로 관찰하는 법"으로 연결한다.
7. 마지막으로 L1/L2 비교를 결제 서비스 관점으로 다시 묶는다.

실습 질문은 이렇게 구성한다.

| 질문 | 학생이 찾아야 할 답 |
| --- | --- |
| USDC를 갖고 있어도 ETH가 없으면 왜 전송이 막힐 수 있는가? | gas token이 별도이기 때문 |
| BaseCoin이 gas token이면 무엇이 쉬워지는가? | 수수료를 원화 단위로 예측하기 쉬움 |
| 전용 네트워크는 왜 범용 네트워크와 다른가? | 모든 dApp보다 결제/송금 UX에 최적화하기 때문 |
| L2가 싸고 빠르다면 무조건 L1보다 좋은가? | 보안 의존성, 정산, 출금 시간, 운영 목적을 함께 봐야 함 |
| Explorer에서 무엇을 보면 네트워크 특성을 알 수 있는가? | gas token, tx fee, block time, status, from/to/value |

설문 요약에서는 기초 이론 보강, 환경 세팅과 오류 해결, 단계별 체크리스트가 우선 요구로 나타났다. Week4에서는 네트워크 이름을 외우게 하기보다 "왜 수수료 단위가 UX를 바꾸는가"를 이해시키는 것이 중요하다.

## 자주 헷갈리는 지점

| 헷갈리는 말 | 정확한 구분 |
| --- | --- |
| "스테이블코인 전용 네트워크는 스테이블코인 그 자체다" | 아니다. 스테이블코인은 자산이고, 전용 네트워크는 그 자산을 전송/정산하는 인프라다. |
| "USDC를 보내면 수수료도 자동으로 USDC로 낸다" | 네트워크 설계에 따라 다르다. 일반 이더리움에서는 gas token이 ETH다. |
| "수수료가 안정적이면 무조건 무료다" | 안정적이라는 말은 예측 가능하다는 뜻에 가깝다. 무료, 후원, 저렴함은 네트워크 설계별로 다르다. |
| "L1은 항상 느리고 L2는 항상 빠르다" | 일반화하면 위험하다. 각 네트워크의 합의, 최종성, 혼잡, 브릿지 구조를 봐야 한다. |
| "전용 네트워크는 범용 네트워크보다 항상 우월하다" | 결제/송금에는 유리할 수 있지만, 범용 dApp 생태계와 보안 분산성은 별도 평가가 필요하다. |
| "StableNet 정보를 최신 사실로 외우면 된다" | 이 보강 자료는 기존 수업 자료 기준으로 설명한다. 실제 운영 상태, Chain ID, Faucet 조건은 바뀔 수 있다. |
| "Faucet이 있으면 메인넷 가치가 있는 코인을 받는 것이다" | Faucet은 테스트넷 실습용 토큰 수령 경로다. 실제 가치와 혼동하지 않는다. |
| "원화 페깅이면 가격이 절대 변하지 않는다" | 페깅은 목표 구조다. 담보, 발행자, 상환 구조, 시장 상황에 따라 리스크가 있다. |

스테이블코인 전용 네트워크를 볼 때는 세 층을 나눠야 한다.

| 층 | 질문 | 예 |
| --- | --- | --- |
| 자산 층 | 무엇에 페깅되는가? | USD, KRW, EUR |
| 수수료 층 | gas token은 무엇인가? | ETH, USDC, USDT, BaseCoin, OKRW |
| 네트워크 층 | 어디에서 실행되고 정산되는가? | L1, L2, rollup, 사이드체인 |

이 세 층을 분리하면 "스테이블코인이니까 수수료도 안정적일 것"이라는 오해를 피할 수 있다.

이 MD의 HTML 확장판은 새 HTML을 지금 만들지 않고, 다음 설계를 따라 제작한다.

시뮬레이터 이름: "일반 네트워크 vs 스테이블코인 전용 네트워크 비교"

목적: 사용자가 같은 10달러 또는 10,000원 상당의 송금을 일반 네트워크와 전용 네트워크에서 실행한다고 가정하고, 준비해야 할 토큰, 수수료 예측성, 실패 원인, 확인 시간을 비교한다.

필수 입력 컨트롤:

- 송금 자산 선택: USDT, USDC, BaseCoin, OKRW
- 기준 통화 선택: USD, KRW
- 네트워크 유형 선택: 일반 이더리움형, L2 rollup형, 스테이블코인 전용 L1형
- gas token 선택: ETH, USDC, USDT, BaseCoin, OKRW, 후원 수수료
- ETH 가격 변동 슬라이더: 낮음, 보통, 높음
- 네트워크 혼잡도 슬라이더: 한산, 보통, 혼잡
- 사용자가 가진 잔고 체크박스: 송금 자산 있음, gas token 있음, Faucet 수령 완료
- 초기화 버튼

판정 로직:

```text
1. 송금 자산이 없으면 "보낼 돈 부족"
2. gas token이 별도인데 사용자가 보유하지 않으면 "수수료 토큰 부족"
3. gas token이 변동성 자산이고 혼잡도가 높으면 "수수료 예측 어려움"
4. gas token이 스테이블코인이면 "수수료 단위 이해 쉬움"
5. 전용 네트워크이면 "결제 UX 단순화"를 표시하되, 생태계/보안/운영 리스크도 함께 표시
6. L2이면 "저렴하고 빠를 수 있음"과 "L1 정산/브릿지 의존"을 함께 표시
```

화면 구성:

- 상단: 송금 시나리오 카드 `일반 네트워크`, `L2`, `전용 네트워크`
- 중앙: 비용 계산 표 `보낼 금액`, `필요 gas token`, `예상 수수료`, `실패 가능 원인`
- 오른쪽: UX 판정 `준비물`, `예측성`, `결제 친화성`, `주의점`
- 하단: 기존 자료 연결 버튼 `stablecoin_networks.html 보기`, `stablenet_explorer.html 보기`

실습 연결 문구:

- 일반 네트워크 비교는 `week/week4/stablecoin_networks.html`의 트랜잭션 흐름 시뮬레이션과 연결한다.
- StableNet 탐색은 `week/week4/stablenet_explorer.html`의 Explorer/Faucet 안내와 연결한다.
- 네트워크별 상태와 Chain ID 등은 바뀔 수 있으므로 HTML에는 "기존 수업 자료 기준" 표시와 "실습 당일 공식 화면에서 재확인" 안내를 둔다.

## 체크리스트

- [ ] 스테이블코인을 가격 안정 자산으로 정의할 수 있다.
- [ ] 스테이블코인 전용 네트워크를 결제/송금 UX 최적화 인프라로 설명할 수 있다.
- [ ] 보내는 토큰과 gas token이 다를 수 있음을 설명할 수 있다.
- [ ] ETH gas 모델과 스테이블코인 gas 모델의 사용자 경험 차이를 말할 수 있다.
- [ ] L1과 L2를 보안 의존성, 정산, 수수료 관점에서 비교할 수 있다.
- [ ] StableNet의 BaseCoin 설명을 "기존 수업 자료 기준"으로 제한해서 말할 수 있다.
- [ ] Faucet 토큰과 실제 가치가 있는 메인넷 토큰을 혼동하지 않는다.
- [ ] Explorer에서 tx fee, gas token, from/to/value, status를 확인할 수 있다.
- [ ] 전용 네트워크의 장점뿐 아니라 생태계, 보안, 규제, 운영 리스크도 질문할 수 있다.
