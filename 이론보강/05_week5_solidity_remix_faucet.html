정석 개념 설명

Solidity 컨트랙트의 정의는 EVM에서 실행되는 코드와 블록체인 storage에 저장되는 상태의 묶음이다. 상태 변수는 모든 노드가 복제하는 영구 데이터이므로 쓰기에는 gas가 든다.

require(condition, "message")는 조건이 참일 때만 다음 줄로 진행한다. 조건이 거짓이면 EVM은 REVERT로 실행을 멈추고, 그 트랜잭션 안에서 바뀌던 상태를 원상복구한다.

비유로 먼저 이해하기

컨트랙트는 자동 판매기, Faucet은 무료 음수대, Remix는 설계도 검사와 버튼 테스트를 하는 작업대에 가깝다. 쿨다운이나 잔액 조건을 어기면 버튼은 눌렸지만 물건이나 물은 나오지 않는다.

개념 보강: 비유 → 정석 정의 → 확인 질문

컨트랙트는 자동 판매기, require는 구매 조건 검사, event는 영수증이다. Faucet에서는 `mapping`이 주소별 마지막 수령 시각을 기억하고, `block.timestamp`와 `LOCK_TIME`이 cooldown을 검사하며, `address(this).balance`가 Faucet 물탱크 잔액이다. 확인 질문은 "이 실패가 cooldown인가, Faucet 잔액인가, 내 지갑 gas 문제인가?"이다.

왜 배우는가

Week5의 목표는 Remix 버튼을 많이 눌러보는 것이 아니라 실패 위치를 좁히는 법을 익히는 것이다. tx hash가 없으면 아직 체인 실행 전이고, receipt가 failed이면 지갑 서명 이후 컨트랙트 조건이나 gas/nonce 증거를 봐야 한다.

처음이면 이 순서로

Compile 검사 -> Deploy 검사 -> Function Call 검사 -> Faucet withdraw() -> 오류 분류 순서로 진행한다. 막히면 다음 단계로 넘어가지 않고 지금 실패가 컴파일 전인지, 컨트랙트 주소 생성 전인지, write tx revert인지 먼저 말한다.

Remix 단계 시뮬레이터

왜 해보는가: 학생들은 "오류가 났다"를 한 문장으로 말하지만, 컴파일 실패와 배포 실패와 함수 실행 실패는 확인할 위치가 완전히 다르다. 이 시뮬레이터는 각 단계가 무엇을 검증하는지 분리한다.

막히면 여기로: 단계 판정이 애매하면 아래 시뮬레이터를 건너뛰지 말고 오류 분류 훈련에서 원인 영역을 먼저 고른다.

Read와 Write 구분: Remix의 읽기 함수는 보통 `eth_call`처럼 현재 상태를 로컬로 조회하고 storage를 바꾸지 않는다. 쓰기 함수는 지갑 서명, gas, nonce, require 조건, 트랜잭션 receipt까지 확인해야 하는 실제 상태 변경 요청이다.

예측 → 실행 → 상태 변화 → 해석

버튼을 누르기 전 실패 위치를 예측한다. Compile 실패는 아직 체인에 간 것이 아니고, Deploy 실패는 네트워크/gas 조건, Call 실패는 함수 종류와 require 조건으로 해석한다.

Faucet require 시뮬레이터

왜 해보는가: Faucet 실패는 웹사이트가 고장난 것이 아니라 쿨다운, 컨트랙트 잔액, 지갑 gas, 네트워크 조건 중 하나가 막힌 경우가 많다. 성공할 때만 storage 변경과 event log가 남는다는 점을 직접 확인한다.

막히면 여기로: cooldown이나 Faucet empty도 write transaction에서는 execution reverted처럼 보일 수 있다. 화면 증상만 보지 말고 오류 분류 훈련에서 원인 영역을 다시 나눈다.

예측 → 실행 → 상태 변화 → 해석

withdraw 전에는 쿨다운과 컨트랙트 잔액이 통과할지 예측한다. 성공하면 물방울, 잔액 감소, 이벤트 영수증을 함께 보고, 실패하면 storage와 event가 바뀌지 않는 것으로 해석한다.

Faucet 안전난간

배포할 때 constructor 인자가 있으면 Remix Deploy 입력칸에 먼저 넣고, 실패한 트랜잭션은 Remix 터미널 또는 explorer의 revert reason에서 어떤 require가 막혔는지 확인한다. ETH 전송은 예전 예제의 transfer보다 call{value: amount}("")을 더 자주 보지만, 초심자 단계에서는 먼저 Checks-Effects-Interactions 순서만 기억한다. 즉 쿨다운과 잔액을 검사하고, lastWithdrawalTime[msg.sender]를 먼저 갱신한 뒤, 마지막에 외부 전송을 한다. 이렇게 하면 받는 주소가 다시 withdraw()를 부르는 reentrancy 상황에서도 같은 시간대 중복 수령을 막는 안전난간이 된다.

타당성 기준

성공할 때만 storage 변경과 event log가 남고, cooldown 실패와 Faucet 잔액 부족은 서로 다른 require 실패로 보여야 한다. 이 화면이 모든 오류를 "Faucet 안 됨"으로 뭉개면 부적합하다.

오류 분류 훈련

왜 해보는가: `ParserError`, Faucet empty, cooldown, insufficient funds, nonce too low, execution reverted는 서로 다른 층의 문제다. 한 번에 "Faucet이 안 된다"라고 부르면 원인을 좁힐 수 없다.

분류 기준: 여기서의 분류는 화면에 보인 에러 문자열이 아니라 root cause domain이다. Faucet empty와 cooldown은 원인 영역으로는 Faucet/컨트랙트 조건이지만, MetaMask나 receipt에서는 execution reverted로 보일 수 있다.
