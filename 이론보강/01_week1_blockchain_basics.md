# Week 1. 블록체인 기초: 공유 장부, 해시, 체인

## 학습 목표

- 블록체인을 "여러 사람이 같은 기록을 나누어 보관하는 공유 장부"로 설명할 수 있다.
- 블록, 거래 데이터, 이전 해시, 현재 해시가 왜 함께 저장되는지 이해한다.
- 블록 데이터가 바뀌면 해시가 바뀌고, 그 결과 다음 블록과의 연결이 깨지는 원리를 설명할 수 있다.
- Week1 가격 조회 과제의 `API 호출 -> 데이터 파싱 -> 화면 렌더링` 흐름을 블록체인 학습의 출발점과 연결한다.
- 기존 `blockchain-interactive-guide.html`의 블록 생성, 해시, 위변조 탐지 시뮬레이터를 이론 복습 자료로 활용한다.

## 비유로 먼저 이해하기

블록체인은 반 학생들이 함께 쓰는 공동 출석부와 비슷하다. 한 명만 출석부를 들고 있으면 그 사람이 실수하거나 몰래 고쳐도 다른 사람이 바로 알기 어렵다. 그런데 모든 학생이 같은 출석부 사본을 가지고 있고, 새 기록이 생길 때마다 서로 비교한다면 한 사람의 기록만 몰래 바꾸기는 매우 어렵다.

블록은 출석부의 한 페이지다. 각 페이지에는 그날의 기록이 들어가고, 맨 아래에는 그 페이지 내용을 요약한 특수한 지문 같은 값이 붙는다. 이 지문이 해시다. 페이지의 글자 하나만 바뀌어도 지문이 완전히 달라진다.

체인은 각 페이지가 이전 페이지의 지문을 함께 적어 두는 방식이다. 3페이지가 2페이지의 지문을 기억하고 있는데, 누군가 2페이지 내용을 바꾸면 2페이지 지문이 바뀐다. 그러면 3페이지에 적힌 "이전 페이지 지문"과 맞지 않으므로 연결이 끊어진다. 그래서 블록체인은 "수정이 절대 불가능한 마법 파일"이라기보다, 수정 시도가 매우 빨리 드러나는 구조에 가깝다.

Week1의 CoinGecko 가격 조회 과제도 이 출발점과 연결된다. 가격 앱은 외부 API에서 데이터를 가져와 화면에 보여준다. 블록체인도 결국 데이터를 읽고, 검증하고, 화면이나 탐색기에서 확인하는 시스템이다. 차이는 CoinGecko 가격 데이터는 특정 API 제공자가 응답하지만, 블록체인 거래 기록은 여러 노드가 같은 장부 상태에 합의한 결과라는 점이다.

## 정석 개념 설명

블록체인은 거래나 상태 변경 기록을 블록 단위로 묶고, 각 블록을 암호학적 해시로 연결한 분산 원장 구조다. 핵심은 기록 저장 방식, 변경 탐지 방식, 네트워크 합의 방식이 함께 작동한다는 점이다.

| 개념 | 정석 설명 | 초심자용 핵심 |
| --- | --- | --- |
| 원장 | 거래 기록을 시간 순서대로 저장한 장부 | 누가 누구에게 무엇을 보냈는지 적은 기록 |
| 분산 원장 | 여러 노드가 같은 원장 사본을 보관하는 구조 | 한 서버가 아니라 여러 컴퓨터가 같이 보관 |
| 블록 | 일정 시간 또는 조건에 따라 묶인 거래 데이터 단위 | 장부의 한 페이지 |
| 해시 | 임의 길이 입력을 고정 길이 값으로 바꾸는 함수 결과 | 데이터 지문 |
| 이전 해시 | 이전 블록의 해시를 현재 블록에 저장한 값 | 앞 페이지와 이어지는 고리 |
| 체인 | 블록들이 이전 해시를 통해 순서대로 연결된 구조 | 페이지들이 지문으로 묶인 장부 |
| 위변조 탐지 | 데이터 변경 시 해시 불일치가 발생하는 성질 | 몰래 고치면 연결이 어긋남 |

해시 함수는 같은 입력에는 항상 같은 결과를 내지만, 입력이 조금만 바뀌어도 결과가 크게 달라진다. 또한 해시값만 보고 원본 데이터를 현실적으로 역산하기 어렵다. 블록체인은 이 성질을 이용해 블록 내용을 요약하고, 다음 블록이 이전 블록의 해시를 참조하게 만든다.

블록체인에서 중요한 점은 "데이터가 저장된다"보다 "데이터가 어떤 순서로 저장되었고, 그 순서가 바뀌지 않았음을 어떻게 검증하는가"이다. 블록 안의 거래가 바뀌면 현재 블록의 해시가 바뀐다. 그러면 다음 블록에 저장된 이전 해시와 맞지 않는다. 더 뒤의 블록까지 모두 다시 계산해야 하므로, 여러 노드가 감시하는 공개 네트워크에서는 임의 변경이 매우 어렵다.

중앙화 시스템과 탈중앙화 시스템도 구분해야 한다. 중앙화 시스템에서는 한 기관의 데이터베이스가 원본이다. 탈중앙화 블록체인에서는 네트워크 참여자들이 정해진 규칙으로 올바른 기록을 선택한다. 그래서 블록체인은 단순 데이터베이스가 아니라, 기록과 검증 규칙과 네트워크 합의가 결합된 시스템이다.

초심자용 시뮬레이터에서는 여러 노드가 같은 기록을 갖는지 비교하지만, 실제 블록체인 합의는 단순한 노드 수 투표가 아니다. 네트워크마다 stake, hash power, validator set, fork choice, finality 같은 규칙으로 "어떤 기록을 정식 체인으로 인정할지"를 정한다. 따라서 로컬에서 해시를 다시 맞추는 것과 네트워크가 그 기록을 최종 기록으로 인정하는 것은 다른 문제다.

## 수업 실습과 연결

Week1 실습은 `week/week_1_Blockchain Lab Report`의 CoinGecko 가격 조회 앱이었다. 이 앱은 `crypto-prices.html`에서 Bitcoin과 Ethereum 가격을 API로 받아 표에 표시하고, `presentation.html`은 `fetchPrices()`, `tryFetch()`, CORS 프록시, 30초 자동 갱신, HTTP 429 Rate Limit 문제를 설명한다.

이 실습은 직접 블록을 만들지는 않지만, 블록체인 실습에 필요한 첫 번째 감각을 준다.

| Week1 실습 요소 | 블록체인 개념과의 연결 |
| --- | --- |
| `fetch()`로 외부 API 호출 | 블록체인 앱도 RPC나 explorer API로 네트워크 데이터를 읽는다. |
| JSON 응답 파싱 | 블록, 트랜잭션, 잔액도 구조화된 데이터로 읽는다. |
| 응답 상태 확인 | 블록체인에서도 성공, 실패, pending, revert 같은 상태를 확인해야 한다. |
| 30초 자동 갱신 | 블록체인 상태도 새 블록 생성에 따라 계속 변한다. |
| CORS와 프록시 | 브라우저 보안 정책과 서버 경유 구조를 이해하면 DApp 구조를 이해하기 쉽다. |
| HTTP 429 Rate Limit | 외부 서비스 호출 한도는 Faucet, RPC, explorer에서도 반복해서 만난다. |

기존 `blockchain-interactive-guide.html`은 Week1 이론 복습용으로 바로 연결할 수 있다. 특히 블록체인 생성, P2P 네트워크 전파, 해시 함수 체험, 합의 알고리즘, 위변조 탐지 시뮬레이션은 Week1 보강 HTML에서 링크하거나 축약 활용하기 좋다.

수업 흐름으로는 먼저 가격 조회 앱에서 "외부 데이터를 가져와 화면에 보여주는 흐름"을 확인한다. 그 다음 블록체인에서는 가격이 아니라 블록 번호, 거래 해시, 주소 잔액, 트랜잭션 상태를 읽는다고 연결한다. 마지막으로 블록 데이터가 해시로 묶여 있어 단순 API 데이터보다 검증 구조가 강하다는 점을 설명한다.

## 자주 헷갈리는 지점

| 헷갈리는 말 | 바로잡기 |
| --- | --- |
| 블록체인은 모든 데이터를 저장한다 | 모든 데이터를 저장하는 것이 아니라, 네트워크 규칙에 따라 필요한 거래와 상태 변경 기록을 저장한다. |
| 블록체인은 수정이 절대 불가능하다 | 물리적으로 수정 시도는 가능하지만, 해시 연결과 네트워크 검증 때문에 정상 기록으로 인정받기 어렵다. |
| 해시는 암호화와 같다 | 해시는 복호화가 없는 요약 함수다. 암호화는 키가 있으면 원문 복원이 가능하지만, 해시는 원문 복원을 목표로 하지 않는다. |
| 블록 하나만 바꾸면 끝이다 | 현재 블록 해시가 바뀌면 다음 블록의 이전 해시와 맞지 않아 뒤 체인 전체 문제가 된다. |
| CoinGecko API 데이터가 블록체인 데이터다 | CoinGecko는 가격 정보 API다. 블록체인 자체 기록은 노드 RPC나 explorer에서 확인한다. |
| API 오류는 블록체인 오류다 | Week1의 CORS, 429는 웹/API 문제다. 블록체인 거래 실패와 분리해서 봐야 한다. |

초심자는 "데이터 조회 앱"과 "블록체인 네트워크"를 한 덩어리로 이해하기 쉽다. Week1에서는 둘을 연결하되 구분해야 한다. 가격 앱은 블록체인 실습을 위한 웹 데이터 처리 입문이고, 블록체인 원리는 공유 장부, 해시, 체인, 합의에서 시작한다.

## HTML 시뮬레이터 설계

Week1 HTML은 새 자료를 만들 때 다음 시뮬레이터를 포함해야 한다. 이 단계에서는 HTML을 만들지 않고 설계만 고정한다.

### 시뮬레이터 1: 블록 데이터 수정과 해시 변경

- 목적: "데이터 한 글자 변경 -> 해시 변경"을 직접 보게 한다.
- 화면 구성:
  - Block #1, Block #2, Block #3 카드 3개를 가로 또는 세로로 배치한다.
  - 각 블록에는 `data`, `previousHash`, `hash`, `status`를 보여준다.
  - Block #2의 거래 데이터 입력창을 제공한다.
- 인터랙션:
  - 사용자가 Block #2 데이터를 바꾸면 Block #2의 `hash`가 즉시 바뀐다.
  - Block #3의 `previousHash`는 기존 값을 유지하므로 "연결 깨짐" 상태가 표시된다.
  - "체인 다시 계산" 버튼을 누르면 Block #3의 `previousHash`가 새 해시로 갱신되어 로컬 연결 상태로는 정상처럼 보인다.
  - "초기화" 버튼으로 원래 데이터와 해시를 복구한다.
- 핵심 메시지:
  - 블록체인은 변경을 불가능하게 만드는 것이 아니라, 변경 사실을 드러나게 만든다.
  - 로컬 재계산은 구조 복구 예시일 뿐이며, 실제 네트워크의 정식 기록은 합의 규칙과 finality 조건을 통과해야 한다.

### 시뮬레이터 2: 중앙 장부와 분산 장부 비교

- 목적: 한 서버만 원본인 구조와 여러 노드가 같은 장부를 비교하는 구조를 구분하게 한다.
- 화면 구성:
  - 왼쪽은 중앙 서버 1개와 사용자 3명, 오른쪽은 노드 5개가 같은 블록을 들고 있는 그림으로 배치한다.
  - "중앙 서버 데이터 수정" 버튼과 "노드 하나만 데이터 수정" 버튼을 둔다.
- 인터랙션:
  - 중앙 서버 데이터가 바뀌면 사용자 모두가 바뀐 값을 보게 한다.
  - 분산 장부에서 한 노드만 바뀌면 나머지 4개 노드와 불일치 표시를 낸다.
  - 과반수 일치 기록이 정상 기록으로 선택되는 시각 효과를 둔다.
- 핵심 메시지:
  - 분산 구조에서는 한 참여자의 기록만 바뀌어도 전체 기록으로 인정되기 어렵다.

### 시뮬레이터 3: Week1 API 실습과 블록체인 데이터 읽기 연결

- 목적: `fetch()`로 가격을 읽는 흐름과 RPC/explorer로 블록 데이터를 읽는 흐름의 공통점과 차이를 보여준다.
- 화면 구성:
  - `CoinGecko API -> 가격 JSON -> 가격 테이블`
  - `RPC/explorer -> 블록/트랜잭션 JSON -> 블록 정보 카드`
- 인터랙션:
  - "가격 데이터 읽기" 버튼은 BTC/ETH 예시 JSON을 표로 렌더링한다.
  - "블록 데이터 읽기" 버튼은 블록 번호, 거래 수, 이전 해시 예시 JSON을 카드로 렌더링한다.
  - "429 API 제한" 토글을 켜면 가격 데이터 쪽에 호출 제한 오류를 표시하고, 블록체인 실패가 아니라 API 서비스 제한이라고 설명한다.
- 핵심 메시지:
  - 웹 앱 입장에서는 둘 다 데이터를 읽고 표시하지만, 데이터의 신뢰 원천과 검증 방식은 다르다.
