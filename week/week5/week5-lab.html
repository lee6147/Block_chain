컨트랙트를 먼저 배포하세요
좌측 에디터에서 컴파일 → 배포
읽기 전용 (view / pure)
owner() view external
컨트랙트 배포자(소유자)의 주소를 반환합니다.
💡 address payable과 가시성(visibility)
address payable public owner에서:
• payable: 이 주소로 ETH를 전송(.transfer())할 수 있음. 일반 address는 전송 불가.
• public: 자동 getter 함수 생성. external은 외부 호출만 가능(가스 효율적), public은 내부+외부 모두 가능.
• 이 컨트랙트의 withdraw(), deposit() 등은 external — 내부 호출이 불필요하므로 가스를 절약합니다.
WITHDRAWAL_AMOUNT() constant
1회 출금 가능 금액입니다. 코드에서 설정한 값이 적용됩니다.
💡 이벤트(Event)와 indexed
event Withdrawal(address indexed user, uint256 amount);
• emit은 블록체인 로그에만 기록되며 상태(storage)를 변경하지 않습니다. 가스비가 저렴합니다.
• indexed 파라미터는 필터링 가능합니다. DApp에서 특정 사용자의 출금 이벤트만 조회할 수 있습니다.
• 최대 3개의 indexed 파라미터를 사용할 수 있습니다.
LOCK_TIME() constant
출금 사이의 최소 대기시간입니다. (1 days = 86400초)
lastWithdrawalTime(address) view
특정 주소의 마지막 출금 시각(타임스탬프)을 조회합니다.
쓰기 (트랜잭션 발생)
withdraw() external
Faucet에서 WITHDRAWAL_AMOUNT만큼 ETH를 수령합니다. LOCK_TIME에 1회만 가능합니다.
💡 mapping 기본값과 Sybil Attack
lastWithdrawalTime[새주소]는 기본값 0을 반환합니다.
따라서 새 주소는 항상 block.timestamp >= 0 + LOCK_TIME을 만족하여 즉시 출금 가능합니다. (의도된 동작)

⚠️ Sybil Attack: 공격자가 새 지갑 주소를 무한 생성하면 LOCK_TIME 제한을 우회할 수 있습니다. 실무에서는 KYC 인증, 캡챠, IP 제한 등 오프체인 방어가 필요합니다.
deposit() payable onlyOwner
Owner가 Faucet에 ETH를 입금합니다. msg.value만큼 전송됩니다.
💡 modifier와 onlyOwner 설계
modifier의 _; 동작 원리:
modifier onlyOwner() { require(...); _; }
_;는 "이 자리에 원래 함수 본문이 삽입된다"는 의미입니다. require()가 먼저 실행되고, 통과하면 함수 본문이 실행됩니다.

설계 의도: deposit()은 onlyOwner로 제한되어 있지만, receive()는 누구나 ETH를 보낼 수 있습니다. 실제 Faucet은 누구든 기부할 수 있어야 하므로 receive()가 그 역할을 합니다.
withdrawAll() onlyOwner
Owner가 Faucet의 전체 ETH를 회수합니다.
receive() 테스트
직접 ETH 전송 receive()
컨트랙트 주소로 직접 ETH를 보내면 receive() external payable가 호출됩니다. 누구나 전송 가능합니다.

🏆 도전과제
🎯
도전 1: WITHDRAWAL_AMOUNT를 0.2 ether로 변경하고 다시 컴파일/배포하여 출금 금액 변화를 확인해보세요!
uint256 public constant WITHDRAWAL_AMOUNT = 0.2 ether; // 배포 후 withdraw() 실행 시 0.2 ETH가 출금됩니다.
🎯
도전 2: LOCK_TIME을 1 hours(3600초)로 변경하고, 더 짧은 시간에 재출금이 가능한지 확인해보세요!
uint256 public constant LOCK_TIME = 1 hours; // 1시간(3600초) 후 재출금이 가능해집니다. // 기존 24 hours(86400초)보다 훨씬 짧습니다.
🎯
도전 3: receive() 함수를 제거하고 직접 ETH 전송을 시도하면 어떤 에러가 발생하는지 확인해보세요!
// receive() external payable { } 를 삭제하면 // 직접 ETH 전송 시 트랜잭션이 revert 됩니다. // fallback()도 없으면 컨트랙트가 ETH를 받을 수 없습니다. // deposit()은 payable이므로 여전히 동작합니다.
🎯
도전 4: withdraw()에서 lastWithdrawalTime 업데이트와 transfer() 호출 순서를 바꿔보세요. 왜 현재 순서(CEI 패턴)가 더 안전한지 생각해보세요!
// CEI 패턴 (Checks-Effects-Interactions) // ✅ 안전한 순서 (현재 코드): // 1. Checks: require() 검증 // 2. Effects: lastWithdrawalTime[msg.sender] = block.timestamp; // 3. Interactions: payable(msg.sender).transfer(...) // // ❌ 위험한 순서: // 1. Checks: require() 검증 // 2. Interactions: payable(msg.sender).transfer(...) ← 먼저 전송 // 3. Effects: lastWithdrawalTime[msg.sender] = ... ← 나중에 업데이트 // // ⚠️ 현재 코드는 transfer()를 사용하므로 2,300 gas만 전달됩니다. // 이 가스로는 상태 변경 함수를 재호출할 수 없어 재진입이 사실상 불가능합니다. // // 그러나 call{value}()를 사용하면 가스 제한이 없으므로, // 수신자가 악의적 컨트랙트라면 receive()에서 다시 withdraw()를 호출할 수 있습니다. // 상태가 아직 업데이트되지 않았으므로 require()를 통과합니다! // 이것이 "재진입(Reentrancy) 공격"이며, CEI 패턴으로 방어합니다.
🎯
도전 5: transferOwnership(address newOwner) 함수를 추가하여 owner를 변경할 수 있도록 해보세요! 다른 계정으로 전환 후 deposit이 되는지 확인해보세요.
function transferOwnership(address newOwner) external onlyOwner { require(newOwner != address(0), "Invalid address"); owner = payable(newOwner); } // 실무에서는 직접 구현하지 않고 // OpenZeppelin의 Ownable 라이브러리를 사용합니다: // import "@openzeppelin/contracts/access/Ownable.sol"; // contract Faucet is Ownable { ... }
🎯
도전 6: WITHDRAWAL_AMOUNT를 constant 대신 immutable로 바꾸고, constructor에서 초기화해보세요! 두 키워드의 차이를 확인해보세요.
// constant: 컴파일 시점에 값이 결정됨 (코드에 직접 삽입) uint256 public constant WITHDRAWAL_AMOUNT = 0.1 ether; // immutable: 배포 시점(constructor)에서 1회 설정 가능 uint256 public immutable WITHDRAWAL_AMOUNT; constructor(uint256 _amount) { owner = payable(msg.sender); WITHDRAWAL_AMOUNT = _amount; // 배포할 때 정할 수 있음 } // constant → 가스비 가장 저렴 (bytecode에 직접 삽입) // immutable → 가스비 저렴 (storage 읽기 불필요) // 일반 변수 → 가스비 비쌈 (SLOAD 필요)
✍️ 방법: 왼쪽 에디터에서 코드 수정 → 컴파일 → 배포하면 변경된 동작을 확인할 수 있습니다!
💡 재배포해도 실습 가이드 진행률은 유지됩니다. 안심하고 코드를 수정해보세요!
(전체 리셋은 에디터의 "🔄 리셋" 버튼을 사용하세요)
⚠️ 시뮬레이터 안내: 이 시뮬레이터는 WITHDRAWAL_AMOUNT, LOCK_TIME 상수 변경과 receive() 함수 제거만 실시간 반영됩니다. 그 외 로직 변경(함수 추가, require 조건 수정 등)은 실제 Remix IDE에서 테스트해보세요!
🚨 프로덕션 vs 교육용 — 이 코드의 한계
현재 Faucet은 교육용 기본 예제입니다. 실제 운영 환경에서는 다음과 같은 보완이 필요합니다:
⚠️
Sybil Attack 방어: 새 주소를 무한 생성하면 LOCK_TIME 제한을 우회할 수 있습니다. 실무에서는 오프체인 인증(KYC, 캡챠, IP 제한)이 필수입니다.
⚠️
긴급 중단(Pausable): 악용 발견 시 출금을 일시 중단하는 pause()/unpause() 기능이 없습니다. OpenZeppelin의 Pausable 라이브러리를 사용해야 합니다.
⚠️
동적 파라미터: WITHDRAWAL_AMOUNT와 LOCK_TIME이 constant로 고정되어 배포 후 변경 불가합니다. 실무에서는 owner가 setter 함수로 동적 조정해야 합니다.
⚠️
전체 출금 한도(Daily Cap): 개별 LOCK_TIME은 있지만, 컨트랙트 전체의 하루 최대 출금 총량 제한이 없어 빠르게 잔액이 소진될 수 있습니다.
⚠️
잔액 부족 알림: 잔액이 임계값 이하로 떨어졌을 때 event LowBalance(uint256 remaining) 같은 경고 이벤트가 없어 능동적 대응이 어렵습니다.
🚀 고급 도전과제 (Remix IDE에서 도전!)
🏅
고급 1: WITHDRAWAL_AMOUNT와 LOCK_TIME을 constant 대신 일반 상태 변수로 바꾸고, owner만 변경할 수 있는 setWithdrawalAmount(), setLockTime() setter 함수를 추가해보세요.
uint256 public withdrawalAmount = 0.1 ether; // constant 제거 uint256 public lockTime = 1 days; // constant 제거 function setWithdrawalAmount(uint256 _amount) external onlyOwner { require(_amount > 0 && _amount <= 1 ether, "Invalid amount"); withdrawalAmount = _amount; } function setLockTime(uint256 _time) external onlyOwner { require(_time >= 1 hours && _time <= 30 days, "Invalid time"); lockTime = _time; }
🏅
고급 2: OpenZeppelin의 Pausable 패턴을 직접 구현해보세요. bool public paused 상태 변수와 whenNotPaused modifier를 만들어 withdraw()에 적용해보세요.
bool public paused; modifier whenNotPaused() { require(!paused, "Contract is paused"); _; } function pause() external onlyOwner { paused = true; } function unpause() external onlyOwner { paused = false; } function withdraw() external whenNotPaused { // ... 기존 로직 ... }
🏅
고급 3: 하루 최대 출금 총량(dailyCap)을 구현해보세요. mapping(uint256 => uint256)으로 날짜별 출금 총량을 추적하고, 한도를 초과하면 revert하도록 만들어보세요.
uint256 public dailyCap = 10 ether; mapping(uint256 => uint256) public dailyWithdrawn; function _today() internal view returns (uint256) { return block.timestamp / 1 days; } function withdraw() external { // ... 기존 require 검증 ... require( dailyWithdrawn[_today()] + WITHDRAWAL_AMOUNT <= dailyCap, "Daily withdrawal cap reached" ); dailyWithdrawn[_today()] += WITHDRAWAL_AMOUNT; // ... 기존 로직 ... }
📚 고급 도전과제는 이 시뮬레이터에서 동작하지 않습니다. Remix IDE에서 직접 컴파일/배포하여 테스트해보세요!
