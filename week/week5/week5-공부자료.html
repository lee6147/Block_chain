블록체인이란?

💡 비유로 이해하기

블록체인은 모두가 함께 쓰는 공유 장부입니다. 한 명이 기록을 바꾸면 나머지 참여자들이 "아닌데?"라고 거부합니다. 그래서 위조가 불가능합니다. 이 장부에 기록하려면 수수료(가스비)를 내야 합니다.

📚 정석 설명

블록체인은 여러 컴퓨터(노드)가 동일한 거래 기록을 공유하는 분산 원장 기술입니다. 거래 기록은 "블록" 단위로 묶이고, 각 블록이 이전 블록과 연결되어 "체인"을 형성합니다. 한번 기록되면 수정할 수 없습니다(불변성).

이더리움과 ETH

💡 비유로 이해하기

이더리움은 "프로그램을 실행할 수 있는 블록체인"입니다. 비트코인이 계산기라면, 이더리움은 스마트폰 — 앱(스마트 컨트랙트)을 설치하고 실행할 수 있습니다. ETH는 이더리움 네트워크의 화폐로, 수수료를 내거나 송금할 때 사용합니다.

📚 정석 설명

이더리움(Ethereum)은 스마트 컨트랙트를 실행할 수 있는 분산 컴퓨팅 플랫폼입니다. ETH(이더)는 이더리움의 기본 암호화폐로, 1 ETH = 1018 wei (가장 작은 단위)입니다.

스마트 컨트랙트란?

💡 비유로 이해하기

블록체인 위의 자동 실행 프로그램입니다. 일반 앱은 회사 서버에 있어서 회사가 마음대로 바꿀 수 있지만, 스마트 컨트랙트는 블록체인에 올라가면 작성자도 수정할 수 없고, 조건이 맞으면 자동으로 실행됩니다.

📚 정석 설명

스마트 컨트랙트는 블록체인에 배포된 코드와 데이터의 묶음으로, 미리 정해진 조건에 따라 자동으로 실행됩니다. 중개자 없이 신뢰를 보장합니다.

가스비(Gas Fee)란?

💡 비유로 이해하기

택시 미터기와 같습니다. 택시를 타면 거리에 따라 요금이 나오듯, 블록체인에서 코드를 실행하면 연산량에 따라 수수료(가스비)가 부과됩니다. 복잡한 연산 = 긴 거리 = 비싼 요금.

📚 정석 설명

가스(Gas)는 이더리움에서 연산을 수행하는 데 필요한 비용 단위입니다. 모든 트랜잭션은 가스를 소모하며, 사용자가 ETH로 지불합니다. 데이터를 저장(SSTORE)하는 연산이 가장 비쌉니다.

  • view/pure 함수 (읽기만): 가스비 무료 (외부 호출 시)
  • 상태 변경 함수 (쓰기): 가스비 유료 (ETH 지불)

지갑과 주소

💡 비유로 이해하기

주소(Address)는 은행 계좌번호, 개인키(Private Key)는 비밀번호입니다. 주소는 누구에게나 공개해도 되지만, 개인키는 절대 노출하면 안 됩니다. 개인키가 있으면 해당 주소의 모든 자산을 통제할 수 있습니다.

📚 정석 설명

이더리움 주소는 20바이트(40자리 16진수, 예: 0x5B38Da...)로 표현됩니다. EOA(사람이 개인키로 직접 제어하는 일반 지갑 계정)는 개인키로 제어하고, 컨트랙트 계정은 코드로 제어됩니다.

1 Solidity란?

이더리움 스마트 컨트랙트를 작성하는 프로그래밍 언어

💡 비유로 이해하기

Solidity는 자판기의 프로그래밍 언어다.

동전(ETH)을 넣고 버튼(함수)을 누르면 자동으로 음료(결과)가 나오는 자판기(스마트 컨트랙트)를 만드는 언어입니다. 자판기는 한 번 설치하면 사람 없이도 24시간 동작하고, 규칙을 바꾸려면 아예 새 자판기를 설치해야 합니다.

📚 정석 설명

Solidity는 EVM(이더리움 가상머신 — 코드를 실행하는 엔진) 위에서 실행되는 정적 타입(변수의 종류를 미리 정해야 하는 방식) 고급 프로그래밍 언어입니다. 스마트 컨트랙트 작성에 특화되어 있으며, JavaScript, Python, C++에서 영향을 받았습니다.

  • 정적 타입: 변수의 타입을 컴파일(사람이 작성한 코드를 기계가 이해하는 언어로 번역) 시점에 결정 (uint256, string, address 등)
  • 고급 언어: 사람이 읽기 쉬운 코드 → 기계가 읽는 바이트코드(기계가 실행하는 저수준 코드)로 변환
  • EVM 타겟: 이더리움, Polygon, BSC 등 EVM 호환 체인에서 실행

2 Remix IDE 이해

브라우저 기반 Solidity 통합 개발 환경

💡 비유로 이해하기

Remix는 요리사의 주방이다.

🧊 냉장고(파일 탐색기)에서 재료(코드)를 꺼내고, 🧑‍🍳 조리대(에디터)에서 요리(코딩)하고, 🍳 오븐(컴파일러)에서 굽고(컴파일), 🍽 테이블(배포)에 내놓는(배포) 것입니다.

3-1. SPDX 라이선스 식별자

// SPDX-License-Identifier: MIT
💡 비유로 이해하기

책의 저작권 표시. MIT 라이선스는 "누구나 복사/수정 가능, 원저자 이름만 남기세요"라는 뜻입니다. 마치 오픈소스 요리 레시피에 "출처만 밝히면 자유롭게 사용하세요"라고 적는 것과 같습니다.

📚 정석 설명

SPDX(Software Package Data Exchange)는 소프트웨어 라이선스 표준입니다.

  • Solidity 0.6.8부터 도입
  • 없으면 컴파일러가 경고(warning) 표시
  • 주석(comment)이므로 바이트코드에 포함되지 않음
  • 주요 라이선스: MIT, GPL-3.0, UNLICENSED(비공개)

3-2. pragma solidity (버전 지시자)

pragma solidity ^0.8.26;
💡 비유로 이해하기

"이 레시피는 오븐 200도 이상에서만 작동합니다" = "이 코드는 Solidity 0.8.26 이상에서만 작동합니다". 너무 낮은 버전은 기능이 없고, 너무 높은 버전(0.9.x)은 문법이 달라질 수 있어 호환되지 않습니다.

📚 정석 설명

pragma는 컴파일러에게 소스코드 처리 방식을 지시하는 키워드입니다.

표현 의미 호환 범위
^0.8.26 캐럿(^) - 마이너 버전 호환 0.8.26 ≤ v < 0.9.0
>=0.8.0 <0.9.0 범위 지정 0.8.0 ≤ v < 0.9.0
0.8.26 정확히 이 버전만 0.8.26만 허용

0.8.26
0.7.0 0.8.0 0.8.26 0.9.0 0.9.5

3-3. contract 키워드

contract HelloWorld { // 코드와 데이터의 모음 }
💡 비유로 이해하기

contract는 법적 계약서와 같다. 한번 서명(배포)하면 변경 불가능하고, 조건(함수)에 따라 자동으로 실행됩니다. "A가 B에게 돈을 보내면, C에게 물건을 전달한다"는 계약이 코드로 작성되어 블록체인 위에서 자동으로 집행됩니다.

📚 정석 설명

contract는 EVM에서 실행되는 코드와 데이터의 모음입니다.

  • 고유 주소: 배포 시 0x로 시작하는 주소 할당 (사람 지갑과 동일한 형식)
  • 프로그래밍에서 '설계도'와 비슷합니다. 설계도(contract)대로 실제 건물(배포된 컨트랙트)을 짓는 것입니다. (Java·Python의 class를 아신다면 같은 개념) — 상태 변수(속성) + 함수(메서드)
  • 불변성: 배포 후 코드 수정 불가 (업그레이드 패턴은 별도)
  • 자체 잔액: ETH를 보유하고 전송 가능

3-4. 상태 변수 & 데이터 위치 (storage / memory / calldata)

string private _greeting = "Hello, World!"; // storage에 영구 저장 function setGreeting(string memory newGreeting) public { _greeting = newGreeting; // memory -> storage 복사 }
💡 비유로 이해하기

💰 왜 세 가지를 구분할까요? 비용(가스비)이 다르기 때문입니다! 영구 저장은 비싸고, 임시 저장은 저렴합니다.

🛡️
Storage
금고 (영구 보관, 비용 높음). 단, 이 금고는 투명합니다 — 블록체인의 모든 데이터는 누구나 볼 수 있습니다.
📝
Memory
메모장 (임시, 함수 끝나면 소멸)
📦
Calldata
읽기전용 택배 상자 (수정 불가, 최저가)
📚 정석 설명
데이터 위치 지속성 수정 가능 가스비 사용 위치
storage 영구적 (블록체인에 저장) ✅ 가능 🔥 매우 높음 (20,000 gas) 상태 변수
memory 임시 (함수 실행 중만) ✅ 가능 💧 중간 (기본 3 gas, 메모리 확장 시 이차적으로 증가) 함수 매개변수, 지역 변수
calldata 임시 (함수 실행 중만) ❌ 불가 ❄️ 최저 external 함수 매개변수

4-2. CheckNumber.sol 분석

💡 비유로 이해하기

CheckNumber는 숫자 감별사. 숫자를 건네주면 양수인지, 음수인지, 0인지 판별해줍니다. 음수를 건네면 "안 됩니다!"라며 거래 자체를 취소(revert)합니다.

📚 정석 설명

CheckNumber.sol은 pure 함수와 revert를 실제로 사용하는 예제입니다.

  • pure: 상태 변수를 읽지도, 쓰지도 않는 순수 계산 함수
  • revert: 조건 불충족 시 실행을 즉시 중단하고 에러 메시지 반환
  • 입력값에 따라 3가지 분기: 양수 → 문자열 반환, 음수 → revert, 0 → 문자열 반환
CheckNumber.sol 1// SPDX-License-Identifier: MIT 2pragma solidity ^0.8.26; 3 4contract CheckNumber { 5 int public number = 100; 6 7 function check(int x) public pure returns (string memory) { 8 if (x > 0) { 9 return "The number is positive"; 10 } else if (x < 0) { 11 revert("The number is negative"); 12 } 13 return "The number is zero"; 14 } 15}
📚 pure 함수란?
  • check()은 상태 변수 number를 읽지 않음 (파라미터 x만 사용)
  • 같은 입력 → 항상 같은 출력 (순수 함수)
  • 가스비 무료 (외부 호출 시)
📚 revert 동작 원리
  • 음수 입력 → 즉시 실행 중단
  • 모든 상태 변경 롤백
  • 남은 가스 환불 (이미 사용된 가스는 손실)
    ※ 단, check()는 pure 함수이므로 외부 호출 시 가스비 자체가 무료 — revert되어도 실제 가스 소모 없음
  • 에러 메시지 "The number is negative" 반환

5 Faucet.sol 심층 분석

테스트넷 Faucet 스마트 컨트랙트의 모든 것

5-1. Faucet이란?

💡 비유로 이해하기

Faucet은 공원의 무료 음수대. 누구나 물(ETH)을 마실 수 있지만, 한 사람이 독점하지 못하도록 1일 1회 제한이 있습니다. 관리자(owner)만 물탱크를 채우거나 비울 수 있습니다.

📚 정석 설명

Faucet은 테스트넷에서 소량의 ETH를 무료 배포하는 스마트 컨트랙트입니다.

  • 목적: 개발자가 테스트에 필요한 ETH를 무료로 받을 수 있게 함
  • 레이트 리밋: 주소당 24시간에 1회만 출금 가능 (남용 방지)
  • 출금 한도: 1회 출금량 제한 (0.1 ETH, constant)
  • 소유자 관리: 잔액 충전/전액 회수는 owner만 가능

📄 Faucet.sol 전체 코드

Faucet.sol 1// SPDX-License-Identifier: MIT 2pragma solidity ^0.8.26; 3 4contract Faucet { 5 address payable public owner; 6 uint256 public constant WITHDRAWAL_AMOUNT = 0.1 ether; 7 uint256 public constant LOCK_TIME = 1 days; 8 9 mapping(address => uint256) public lastWithdrawalTime; 10 11 event Withdrawal(address indexed user, uint256 amount); 12 13 modifier onlyOwner() { 14 require(msg.sender == owner, "Only the owner can call this function."); 15 _; 16 } 17 18 constructor() { 19 owner = payable(msg.sender); 20 } 21 22 function withdraw() external { 23 require( 24 block.timestamp >= lastWithdrawalTime[msg.sender] + LOCK_TIME, 25 "You must wait 24 hours between withdrawals."); 26 require( 27 address(this).balance >= WITHDRAWAL_AMOUNT, 28 "Insufficient balance in the faucet."); 29 30 lastWithdrawalTime[msg.sender] = block.timestamp; 31 payable(msg.sender).transfer(WITHDRAWAL_AMOUNT); 32 emit Withdrawal(msg.sender, WITHDRAWAL_AMOUNT); 33 } 34 35 function deposit() external payable onlyOwner {} 36 37 function withdrawAll() external onlyOwner { 38 owner.transfer(address(this).balance); 39 } 40 41 receive() external payable {} 42}
💡 실무 참고: .transfer()는 학습용으로 사용합니다. 실제 프로젝트에서는 2300 gas 제한 문제로 .call{value: amount}("") 사용이 권장됩니다.

5-2. 핵심 개념 완전 정복

① address payable

💡 비유

돈을 받을 수 있는 지갑 주소. 일반 address는 신분증(확인용), payable address는 지갑(돈을 받을 수 있음)입니다.

📚 정석

address payable은 .transfer()와 .send() 메서드를 사용할 수 있어 ETH를 수신할 수 있는 주소 타입입니다. 일반 address는 ETH 전송이 불가합니다.

② mapping

💡 비유

전화번호부. 이름(key)을 알면 전화번호(value)를 바로 찾을 수 있습니다. mapping(address => uint256)은 "주소별 숫자를 기록하는 전화번호부"입니다.

📚 정석

mapping은 해시 테이블 기반 key-value 저장소입니다. 항상 storage에 저장되며, 기본값이 자동 설정됩니다(uint는 0, bool은 false, address는 0x0).

🎮 mapping 동작 시각화

주소와 값을 입력하고 "저장"을 누르면 해시 테이블에 기록되는 과정을 볼 수 있습니다. "조회"로 값을 읽어보세요.

Slot Key (address) Value
// 비어있음 (기본값: 0)
🔍 스토리지 슬롯 계산 과정 저장 또는 조회 시 keccak256 해시 과정이 여기에 표시됩니다.

③ event & emit

💡 비유

영수증 발행. 거래가 완료되면 영수증(event)을 발행(emit)하여 기록합니다. 나중에 누구나 이 영수증을 조회할 수 있습니다.

📚 정석

event는 블록체인의 트랜잭션 로그에 기록되는 데이터입니다. indexed(검색 가능 표시) 키워드로 최대 3개의 매개변수를 필터링/검색 가능하게 만들 수 있습니다. 가스비가 storage보다 훨씬 저렴합니다(로그 1개당 ~375 gas).

④ modifier (함수 수정자)

💡 비유

보안 검문소. 함수(건물)에 들어가기 전에 검문소(modifier)를 통과해야 합니다. 통과하지 못하면 함수가 실행되지 않고 거래가 취소됩니다.

📚 정석

modifier는 함수 실행 전/후에 조건을 검사하는 재사용 가능한 코드입니다. _;는 "원래 함수 코드를 여기서 실행하라"는 플레이스홀더입니다.

modifier onlyOwner() { require(msg.sender == owner, "Only owner"); // 검문 _; // _; = "여기에 원래 함수 코드를 실행하라"는 뜻 (플레이스홀더) }

⑤ msg.sender

💡 비유

전화 건 사람의 번호. 누가 이 함수를 호출했는지 알 수 있는 발신자 번호입니다. 위조 불가능합니다.

📚 정석

msg.sender는 현재 함수를 호출한 주소(address)를 나타내는 전역 변수입니다. 외부 계정(EOA)일 수도 있고 다른 컨트랙트일 수도 있습니다.

⚠️ tx.origin vs msg.sender
msg.sender는 직접 호출자 (컨트랙트일 수 있음) → 보안에 안전
tx.origin은 최초 트랜잭션 발신자 (항상 사람) → 피싱 공격에 취약
절대 require(tx.origin == owner) 를 쓰지 마세요! 악의적 컨트랙트가 사용자를 속여 트랜잭션을 보내게 하면, tx.origin은 사용자 주소가 됩니다.

⑤-2 msg.value

💡 비유

택배 상자에 동봉된 현금 금액. 함수를 호출할 때 "이 돈도 같이 보낼게요"라고 함께 전송한 ETH의 양입니다.

📚 정석

msg.value는 함수 호출 시 함께 전송된 ETH의 양(wei 단위)입니다. payable 함수에서만 사용 가능합니다. deposit() 함수가 payable인 이유는 ETH를 받아야 하기 때문입니다.

⑥ block.timestamp

💡 비유

우체국 소인. 편지(트랜잭션)가 처리된 정확한 시각이 자동으로 찍힙니다. 조작이 어렵습니다.

📚 정석

block.timestamp는 현재 블록이 생성된 시간(Unix timestamp, 초 단위)입니다. 검증자(validator)가 수 초 범위에서 조작 가능 (2022년 PoS 전환 이후)하므로 정밀한 타이밍에는 부적합하지만, 24시간 간격 같은 대략적 시간 체크에는 충분합니다.

⑦ receive() 함수

💡 비유

집 현관의 우편함 — 누군가 문을 두드리지 않고(함수 호출 없이) 우편함(receive)에 소포(ETH)를 넣으면 자동으로 받습니다.

📚 정석

receive()는 컨트랙트가 순수 ETH 전송(calldata 없음)을 받았을 때 자동 호출되는 특수 함수입니다. Solidity 0.6.0에서 도입되었으며, 없으면 ETH 수신이 불가합니다.

receive() external payable {} // 순수 ETH 전송을 받을 수 있게 함

⑧ Solidity 내장 단위 (ether / time)

💡 비유

화폐 단위 변환표. 1만원 = 10,000원인 것처럼, Solidity도 ether와 시간에 대한 단위 변환을 내장하고 있습니다.

⚔️ 5-5. 재진입(Reentrancy) 공격과 방어

💡 비유로 이해하기

ATM 반복 출금 사기. ATM이 돈을 내보낸 뒤 잔액을 차감한다면, 잔액이 줄기 전에 다시 출금 버튼을 누를 수 있습니다. 스마트 컨트랙트에서도 ETH를 보낸 뒤 상태를 업데이트하면, 수신자가 receive() 함수에서 다시 withdraw()를 호출하여 반복 출금할 수 있습니다. 이것이 재진입 공격입니다.

📚 정석 설명

재진입(Reentrancy)은 스마트 컨트랙트 보안의 가장 대표적인 취약점입니다. 2016년 The DAO 해킹($60M 피해)의 원인이었습니다.

공격 원리:

  1. 공격자 컨트랙트가 withdraw() 호출
  2. Faucet이 ETH를 전송 (.transfer() 또는 .call())
  3. 공격자의 receive() 함수가 자동 실행됨
  4. receive() 안에서 다시 withdraw() 호출 → 잔액 차감 전이므로 또 성공
  5. 반복하여 컨트랙트의 모든 ETH 탈취

방어법: Checks-Effects-Interactions (CEI) 패턴 — 상태를 먼저 변경한 후 외부 호출

❌ 취약한 코드
// 3. 먼저 전송 (위험!) msg.sender.call{value: amt}(""); // 1. 나중에 상태 변경 balances[msg.sender] -= amt;
✅ 안전한 코드 (CEI)
// 1. Checks (검증) require(balance >= amt); // 2. Effects (상태 변경) balances[msg.sender] -= amt; // 3. Interactions (외부 호출) msg.sender.call{value: amt}("");

🛡️ 5-6. Checks-Effects-Interactions (CEI) 패턴

💡 비유로 이해하기

은행 출금 3단계. 1) 신분증 확인(Checks) → 2) 장부에 출금 기록(Effects) → 3) 현금 전달(Interactions). 현금을 먼저 주고 나중에 장부를 수정하면 문제가 생깁니다.

📚 정석 설명

CEI 패턴은 스마트 컨트랙트 함수의 코드를 3단계로 구성하는 보안 설계 원칙입니다.

우리의 Faucet.sol withdraw() 함수가 이미 CEI를 따르고 있습니다:

// 1. Checks (검증)
require(block.timestamp >= lastWithdrawalTime[msg.sender] + LOCK_TIME, ...);
require(address(this).balance >= WITHDRAWAL_AMOUNT, ...);
// 2. Effects (상태 변경) ← 외부 호출보다 먼저!
lastWithdrawalTime[msg.sender] = block.timestamp;
// 3. Interactions (외부 호출) ← 가장 마지막!
payable(msg.sender).transfer(WITHDRAWAL_AMOUNT);
emit Withdrawal(msg.sender, WITHDRAWAL_AMOUNT);

6 require vs assert vs revert

에러 처리의 삼총사

💡 비유로 이해하기

식당에서의 비유:

  • require = 입구 안내원: "예약하셨나요? 아니면 돌아가세요" (입력 검증)
  • assert = 비상벨: "주방에 불이 났다! 이건 있을 수 없는 일!" (내부 불변 조건)
  • revert = 매니저 판단: "복잡한 상황을 종합적으로 봤을 때, 이 주문은 취소합니다" (복잡한 조건)

🟢 require

용도: 입력값/외부 조건 검증

require(msg.value > 0, "Send ETH!"); require(balance >= amount, "Not enough");
  • 조건이 false면 revert
  • 남은 가스 환불
  • 에러 메시지 제공 가능
  • 가장 자주 사용

🔴 assert

용도: 내부 불변 조건 검증

assert(totalSupply >= 0); assert(balance[msg.sender] <= totalSupply);
  • 조건이 false면 Panic 에러
  • 0.8.0 이후 가스 환불
  • 에러 메시지 없음
  • 버그 탐지용

🟠 revert

용도: 복잡한 조건부 에러 처리

if (amount > limit) { revert("Over limit"); } revert CustomError(amount);
  • 무조건 revert
  • 남은 가스 환불
  • Custom Error 사용 가능 (가스 절약)
  • 복잡한 로직용
