1

ERC20이란?

비유로 이해하기

USB 포트를 생각해 보세요. 예전에는 제조사마다 충전 포트가 달라서 케이블을 여러 개 들고 다녀야 했습니다. 그런데 USB-C라는 표준이 생기면서 하나의 케이블로 모든 기기를 충전할 수 있게 되었습니다. ERC20은 이더리움 토큰의 USB-C 표준입니다.

  • ● 표준이 없으면 지갑마다 토큰을 따로 지원해야 해서 개발자와 사용자 모두 힘들어집니다.
  • ● ERC20 표준을 따르면 MetaMask, Uniswap, 거래소 등 모든 곳에서 자동으로 호환됩니다.
  • ● ERC = Ethereum Request for Comments. EIP(Ethereum Improvement Proposal, 이더리움 개선 제안)의 하위 카테고리로, 애플리케이션 수준 표준을 정의합니다.

기술적 정의

ERC20은 이더리움 블록체인 위에서 대체 가능한 토큰(Fungible Token)을 구현하기 위한 스마트 컨트랙트 인터페이스 표준입니다. EIP-20으로 공식 등록되어 있습니다.

  • 대체 가능성(Fungibility): 동일한 토큰 간에 가치가 동일하여 교환 가능
  • 표준 인터페이스: 6개의 필수 함수 + 2개의 이벤트로 구성
  • 호환성: 표준을 따르는 모든 DApp이 별도 구현 없이 토큰 연동 가능
// ERC20 표준 인터페이스
interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
    function allowance(address owner, address spender) external view returns (uint256);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);

    event Transfer(address indexed from, address indexed to, uint256 value);
    event Approval(address indexed owner, address indexed spender, uint256 value);
}

핵심 함수 6개

totalSupply()

전체 발행량 조회

비유: 한국은행이 발행한 총 화폐량을 조회하는 것과 같습니다. 시장에 존재하는 토큰 총량을 알 수 있습니다.

balanceOf(address)

잔액 조회

비유: 은행 앱에서 내 잔고를 확인하는 것과 같습니다. 특정 지갑 주소의 토큰 잔액을 반환합니다.

transfer(to, amount)

직접 전송

비유: 인터넷 뱅킹으로 직접 계좌이체하는 것과 같습니다. 본인이 직접 다른 주소로 토큰을 보냅니다.

approve(spender, amount)

사용 권한 부여

비유: 자동이체를 등록하는 것과 같습니다. 특정 컨트랙트가 내 토큰을 대신 사용할 수 있도록 허용합니다.

allowance(owner, spender)

승인량 조회

비유: 자동이체 한도를 확인하는 것과 같습니다. spender가 owner 대신 사용할 수 있는 잔여 허용량을 반환합니다.

transferFrom(from, to, amount)

위임 전송

비유: 자동이체가 실행되는 것과 같습니다. approve로 허가받은 spender가 from의 토큰을 to에게 전송합니다.

// 총 발행량 반환 (읽기 전용)
function totalSupply() external view returns (uint256);

// 특정 주소의 잔액 반환
function balanceOf(address account) external view returns (uint256);

// 호출자(msg.sender)에서 to로 amount만큼 전송
// 성공 시 true 반환, 잔액 부족 시 revert
function transfer(address to, uint256 amount) external returns (bool);

// spender가 호출자 대신 사용할 수 있는 한도 설정
// 덮어쓰기 방식 (이전 값 무시)
function approve(address spender, uint256 amount) external returns (bool);

// owner가 spender에게 허용한 잔여 한도 조회
function allowance(address owner, address spender) external view returns (uint256);

// spender가 from의 토큰을 to로 전송 (allowance 차감)
// DeFi 프로토콜의 핵심 메커니즘
function transferFrom(
    address from,
    address to,
    uint256 amount
) external returns (bool);

이벤트 (Transfer, Approval)

비유로 이해하기

이벤트는 은행 거래 내역서와 같습니다. 돈이 오갈 때마다 기록이 남듯이, 토큰이 전송되거나 승인될 때마다 블록체인에 로그가 기록됩니다. MetaMask나 DApp이 이 로그를 읽어 화면을 실시간으로 업데이트합니다.

Transfer 이벤트

토큰이 전송될 때마다 자동으로 기록됩니다.

  • · from: 보낸 주소
  • · to: 받는 주소
  • · value: 전송량

Approval 이벤트

approve() 호출 시 자동으로 기록됩니다.

  • · owner: 허용한 주소
  • · spender: 허용받은 주소
  • · value: 허용량

활용: 프론트엔드에서 contract.on("Transfer", callback)으로 이벤트를 구독하면 전송이 발생할 때마다 UI를 자동으로 업데이트할 수 있습니다.

이벤트는 EVM 로그(Log)로 저장되며, 스토리지보다 훨씬 저렴합니다. indexed 키워드가 붙은 파라미터는 블룸 필터로 빠르게 검색할 수 있습니다. 

// 토큰 전송 시 발생하는 이벤트 (mint: from=0x0, burn: to=0x0)
event Transfer(address indexed from, address indexed to, uint256 value);

// approve() 호출 시 발생하는 이벤트
event Approval(address indexed owner, address indexed spender, uint256 value);

// ethers.js에서 이벤트 구독 예시
contract.on("Transfer", (from, to, value, event) => {
  console.log(`${from} -> ${to}: ${ethers.formatEther(value)} MTK`);
  // UI 잔액 업데이트 로직 실행
});

전체 코드 해설

비유로 이해하기

이 코드는 회사 설립 서류와 같습니다. 회사를 세울 때 정관(사업 목적, 상호명, 자본금)을 작성하듯이, 스마트 컨트랙트 배포 시 토큰의 이름·심볼·초기 발행량을 설정합니다. 기본적으로 한 번 배포하면 변경할 수 없습니다. (단, Proxy 패턴(UUPS, Transparent Proxy 등)을 사용하면 업그레이드 가능한 컨트랙트를 만들 수 있습니다.)

각 줄이 무엇을 의미하는지 쉽게 설명합니다.

1번 줄

라이선스 표시 — 이 코드는 MIT 라이선스로 누구나 사용 가능합니다.

2번 줄

솔리디티 버전 — 0.8.20 이상 컴파일러를 사용해야 합니다.

3번 줄

OpenZeppelin 라이브러리에서 이미 만들어진 ERC20 코드를 가져옵니다. 우리가 직접 6개 함수를 짤 필요가 없습니다!

4번 줄

MyToken이라는 컨트랙트를 ERC20을 상속받아 만듭니다. (상속 = 기존 기능 전부 가져오기)

5번 줄

배포 시 한 번만 실행되는 초기화 함수 — 이름은 "My Token", 심볼은 "MTK"로 설정합니다.

6번 줄

배포자(msg.sender)에게 initialSupply * 10^18개의 토큰을 발행합니다. (18자리는 소수점 단위. 예: initialSupply=1000이면 내부적으로 1000 * 10^18 = 1,000,000,000,000,000,000,000 최소 단위로 저장됩니다. UI에서는 ethers.formatUnits()로 사람이 읽을 수 있는 "1000"으로 변환합니다.)

// SPDX-License-Identifier: MIT
// ↑ 라이선스 식별자: 컴파일러 경고 방지 및 오픈소스 라이선스 명시

pragma solidity ^0.8.20;
// ↑ 0.8.20 이상 버전에서만 컴파일 허용
//   참고: 오버플로우/언더플로우 자동 체크는 Solidity 0.8.0부터 도입된 기능입니다

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
// ↑ OpenZeppelin의 검증된 ERC20 구현체 임포트
//   totalSupply, balanceOf, transfer 등 모든 표준 함수가 구현되어 있음

contract MyToken is ERC20 {
// ↑ ERC20을 상속받아 모든 표준 기능을 자동으로 획득

    constructor(uint256 initialSupply) ERC20("My Token", "MTK") {
    // ↑ 배포 시 1회 실행되는 생성자
    //   ERC20("My Token", "MTK"): 토큰 이름과 심볼 초기화

        _mint(msg.sender, initialSupply * 10 ** decimals());
        // ↑ _mint: 내부 함수로 새 토큰을 생성하여 msg.sender에게 지급
        //   decimals()는 기본값 18 반환 → 1 토큰 = 1 * 10^18 최소 단위
        //   예: initialSupply=1000 → 1000 * 10^18 wei 단위 토큰 발행
    }
}

핵심 포인트: decimals()가 18을 반환하는 것은 ETH의 최소 단위(Wei, 10^18)와 일관성을 유지하기 위한 관례적 기본값입니다. 필수는 아니며, 토큰마다 자유롭게 설정할 수 있습니다 (예: USDT는 6, WBTC는 8). UI에서는 ethers.formatUnits(balance, decimals)로 읽기 좋은 형태로 변환합니다.

MyToken.sol 전체 코드

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
contract MyToken is ERC20 {
    constructor(uint256 initialSupply) ERC20("My Token", "MTK") {
        _mint(msg.sender, initialSupply * 10 ** decimals());
    }
}

확장 기능 (mint, burn, pause)

mint (추가 발행)

중앙은행이 화폐를 추가로 발행하는 것과 같습니다. 새 토큰을 만들어 특정 주소에 지급합니다.

burn (소각)

화폐를 없애는 것과 같습니다. 토큰을 영구적으로 삭제하여 총 발행량을 줄입니다. 디플레이션 효과.

pause (일시정지)

긴급 상황에서 모든 거래를 멈추는 비상 브레이크입니다. 해킹이나 버그 발견 시 사용합니다.

Ownable 접근 제어: mint와 pause 같은 강력한 기능은 컨트랙트 소유자(배포자)만 사용할 수 있도록 제한합니다. 자물쇠가 달린 관리자 전용 기능이라고 생각하세요. 

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Burnable.sol";
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Pausable.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract MyTokenExtended is ERC20, ERC20Burnable, ERC20Pausable, Ownable {
    constructor(uint256 initialSupply)
        ERC20("My Token", "MTK")
        Ownable(msg.sender)  // 배포자를 소유자로 설정
    {
        _mint(msg.sender, initialSupply * 10 ** decimals());
    }

    // onlyOwner: 소유자만 호출 가능한 mint 함수
    function mint(address to, uint256 amount) public onlyOwner {
        _mint(to, amount);
    }

    // onlyOwner: 비상 시 모든 전송 정지
    function pause() public onlyOwner {
        _pause();
    }

    // onlyOwner: 정지 해제
    function unpause() public onlyOwner {
        _unpause();
    }

    // ERC20Burnable을 통해 burn(), burnFrom() 자동 제공

    // Pausable 충돌 해결을 위한 override 필요
    function _update(address from, address to, uint256 value)
        internal
        override(ERC20, ERC20Pausable)
    {
        super._update(from, to, value);
    }
}

Ownable 소유권 관리

Ownable 패턴은 컨트랙트의 관리자 권한을 제어합니다. 두 가지 중요한 함수가 있습니다:

  • · transferOwnership(newOwner): 소유권을 다른 주소로 이전합니다. 실수로 잘못된 주소에 전달하면 소유권을 되찾을 수 없습니다.
  • · renounceOwnership(): 소유권을 영구히 포기합니다. 호출 후에는 mint, pause 등 onlyOwner 함수를 영영 호출할 수 없습니다.

주의: renounceOwnership()은 탈중앙화를 위해 의도적으로 사용할 수 있지만, 실수로 호출하면 컨트랙트의 관리 기능이 영구 잠금됩니다. 신중하게 사용하세요.

ERC20Capped — 최대 발행량 제한

기본 MyToken은 mint에 상한선이 없어 무한 발행이 가능합니다. ERC20Capped를 사용하면 최대 발행량(cap)을 지정하여 비트코인처럼 총 발행량을 제한할 수 있습니다. 

import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Capped.sol";

contract CappedToken is ERC20Capped, Ownable {
    constructor() ERC20("Capped", "CAP")
        ERC20Capped(1_000_000 * 10**18) // 최대 100만 개
        Ownable(msg.sender) {
        _mint(msg.sender, 500_000 * 10**18); // 초기 50만 개
    }
    function mint(address to, uint256 amount) public onlyOwner {
        _mint(to, amount); // cap 초과 시 자동 revert
    }
    // ERC20 + ERC20Capped 상속 충돌 해결
    function _update(address from, address to, uint256 value)
        internal override(ERC20, ERC20Capped) {
        super._update(from, to, value);
    }
}

EIP-2612 Permit — 가스 없는 승인

일반 approve()는 별도 트랜잭션(가스비)이 필요합니다. EIP-2612 permit()은 오프체인 서명(ECDSA)으로 approve를 대체하여:

  • · 사용자가 가스비 없이 승인할 수 있음 (서명만 하면 됨)
  • · approve + transferFrom을 단일 트랜잭션으로 합칠 수 있음
  • · 온체인 approve 트랜잭션을 제거하고, nonce로 재사용을 방지하며, deadline으로 시간을 제한하여 프론트러닝 위험을 크게 완화
  • · 주의: permit 서명도 mempool에서 추출될 수 있으므로 완벽한 방지는 아닙니다
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Permit.sol";

contract PermitToken is ERC20, ERC20Permit {
    constructor() ERC20("Permit", "PMT") ERC20Permit("Permit") {
        _mint(msg.sender, 1000 * 10**18);
    }
}
// 프론트엔드: permit() 서명 → 릴레이어가 대신 트랜잭션 제출

보안 주의사항

ETH 전송 주의: receive() / fallback() 함수

ERC20 컨트랙트에는 기본적으로 receive()나 fallback() 함수가 없습니다. 실수로 ETH를 ERC20 컨트랙트 주소로 직접 전송하면 영구적으로 잠기게 됩니다. 이는 초보자가 자주 겪는 실수이므로 특히 주의하세요.

  • · ETH를 전송할 때는 반드시 지갑 주소(EOA)로 보내세요.
  • · ERC20 토큰을 전송할 때는 transfer() 함수를 사용하세요.
  • · 컨트랙트에 ETH 수신 기능이 필요하면 receive() external payable를 명시적으로 구현하세요.

배포 전 보안 체크리스트

  • ● Owner 키 관리: 컨트랙트 Owner의 개인키를 안전하게 보관하세요. 분실 시 mint, pause 등의 관리 기능을 영영 사용할 수 없습니다.
  • ● 무제한 mint 위험: MyTokenExtended의 mint 함수에는 발행량 상한이 없습니다. 프로덕션에서는 ERC20Capped로 MAX_SUPPLY를 설정하거나, 타임락/멀티시그를 적용하세요.
  • ● 소스코드 검증: 배포 후 반드시 Etherscan에서 컨트랙트 소스코드를 검증(Verify)하세요. 미검증 컨트랙트는 사용자가 코드를 확인할 수 없어 신뢰도가 떨어집니다.
  • ● 테스트: 배포 전 Hardhat이나 Foundry로 단위 테스트를 작성하여 transfer 성공/실패, 권한 검증 등을 확인하세요.

MetaMask 지갑과 Remix IDE를 이용하여 ERC20 토큰을 실제 테스트넷에 배포합니다. 순서대로 따라해 주세요.

1

Remix IDE 접속

브라우저에서 Remix IDE에 접속합니다. 별도 설치 없이 웹 브라우저만으로 스마트 컨트랙트를 개발하고 배포할 수 있습니다.

Remix IDE 열기 (새 탭)

팁: Chrome 또는 Firefox 브라우저를 사용하세요. MetaMask 확장 프로그램이 설치되어 있어야 합니다.

2

새 파일 생성 → MyToken.sol 코드 붙여넣기

  1. 1. 좌측 파일 탐색기에서 contracts 폴더를 우클릭합니다.
  2. 2. New File을 클릭하고 파일명을 MyToken.sol로 입력합니다.
  3. 3. 아래 코드를 전체 복사하여 편집기에 붙여넣습니다. 
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MyToken is ERC20 {
    constructor(uint256 initialSupply) ERC20("My Token", "MTK") {
        _mint(msg.sender, initialSupply * 10 ** decimals());
    }
}

주의: Remix는 기본적으로 OpenZeppelin 라이브러리를 자동으로 불러올 수 있습니다. 만약 import 오류가 발생하면 DEPS → Install 버튼을 클릭하거나, 좌측 Plugin Manager에서 DEPS 플러그인을 활성화하세요.

3

컴파일러 설정

  1. 1. 좌측 사이드바에서 Solidity Compiler 아이콘(▲ 모양)을 클릭합니다.
  2. 2. COMPILER 드롭다운에서 0.8.20 이상 버전을 선택합니다.
  3. 3. Compile MyToken.sol 버튼을 클릭합니다.
  4. 4. 컴파일 성공 시 아이콘에 초록색 체크 표시가 나타납니다.

팁: Auto compile 체크박스를 활성화하면 코드 수정 시 자동으로 컴파일됩니다. Enable optimization도 체크하면 가스 비용을 줄일 수 있습니다.

4

배포 환경: Injected Provider - MetaMask 선택

  1. 1. 좌측 사이드바에서 Deploy & Run Transactions 아이콘(▶ 모양)을 클릭합니다.
  2. 2. ENVIRONMENT 드롭다운에서 Injected Provider - MetaMask를 선택합니다.
  3. 3. MetaMask 팝업이 열리면 연결을 클릭합니다.
  4. 4. MetaMask에서 테스트넷(Sepolia 등)이 선택되어 있는지 확인합니다.

주의: 실제 이더(ETH)를 사용하지 않으려면 반드시 테스트넷(Sepolia)을 사용하세요. 테스트넷 ETH는 SepoliaFaucet에서 무료로 받을 수 있습니다.

5

생성자 인자 입력 → Deploy

  1. 1. CONTRACT 드롭다운에서 MyToken이 선택되어 있는지 확인합니다.
  2. 2. Deploy 버튼 옆 입력란에 초기 발행량을 입력합니다. 1000 입력 시 1,000 MTK가 발행됩니다.
  3. 3. Deploy 버튼을 클릭합니다.
  4. 4. MetaMask 팝업에서 가스 비용을 확인하고 확인을 클릭합니다.

// 생성자 인자 예시

initialSupply: 1000

// → 배포자 지갑에 1000 MTK (1000 * 10^18 wei) 발행

팁: 트랜잭션이 제출되면 MetaMask에서 Activity 탭에서 진행 상황을 확인할 수 있습니다. 보통 10~30초 내에 완료됩니다.

6

배포된 컨트랙트 주소 확인

  1. 1. Remix 화면 하단 콘솔(Terminal)에서 트랜잭션 결과를 확인합니다.
  2. 2. 배포 성공 후 Deployed Contracts 섹션에 컨트랙트가 나타납니다.
  3. 3. 컨트랙트 이름 옆 복사 아이콘을 클릭하여 주소를 복사합니다.
  4. 4. 아래 저장 폼에 붙여넣기 하세요.

주의: 컨트랙트 주소는 0x로 시작하는 42자리 16진수입니다. 주소를 잘못 저장하면 토큰을 다시 찾을 수 없으니 반드시 정확히 복사하세요.

7

MetaMask에서 "토큰 가져오기" → 주소 입력

  1. 1. MetaMask를 열고 Tokens 탭으로 이동합니다.
  2. 2. 하단의 토큰 가져오기(Import tokens)를 클릭합니다.
  3. 3. 토큰 계약 주소 필드에 6단계에서 복사한 컨트랙트 주소를 붙여넣습니다.
  4. 4. 토큰 심볼(MTK)과 소수점(18)이 자동으로 입력됩니다.
  5. 5. 토큰 추가를 클릭하면 MetaMask 지갑에서 MTK 잔액을 확인할 수 있습니다.

팁: 토큰이 자동으로 감지되지 않으면 Custom Token 탭에서 컨트랙트 주소를 직접 입력하세요.

8

Etherscan에서 소스코드 검증 (Verify)

배포된 컨트랙트의 소스코드를 Etherscan에 등록하면 누구나 코드를 열람할 수 있어 투명성과 신뢰도가 높아집니다. 검증되지 않은 컨트랙트는 바이트코드만 표시되어 사용자가 기능을 확인할 수 없습니다.

  1. 1. sepolia.etherscan.io에 접속하여 배포한 컨트랙트 주소를 검색합니다.
  2. 2. Contract 탭 → Verify and Publish 링크를 클릭합니다.
  3. 3. 컴파일러 버전(0.8.20), 라이선스(MIT)를 선택하고 소스코드를 붙여넣습니다.
  4. 4. Verify and Publish 버튼을 클릭하면 검증이 완료됩니다.

팁: Remix IDE의 Etherscan - Contract Verification 플러그인을 사용하면 IDE에서 바로 검증할 수 있습니다. Hardhat/Foundry를 사용하는 경우 npx hardhat verify 또는 forge verify-contract 명령으로 자동 검증이 가능합니다.

참고: 로컬 개발 환경으로 배포하기

Remix IDE 외에도 실무에서 널리 사용되는 로컬 개발 도구가 있습니다. 프로젝트 규모가 커지면 아래 도구 중 하나를 사용하는 것을 권장합니다.

Hardhat

JavaScript/TypeScript 기반 개발 환경. 테스트, 디버깅, 배포 자동화를 지원합니다.

npx hardhat init npx hardhat compile npx hardhat run scripts/deploy.js --network sepolia
Foundry

Rust 기반 고속 도구. Solidity로 테스트를 작성할 수 있어 가스 최적화에 유리합니다.

forge init forge build forge create src/MyToken.sol:MyToken --rpc-url $SEPOLIA_RPC

보안 주의: approve() 프론트러닝 취약점

allowance를 A → B로 변경할 때, spender가 변경 트랜잭션을 감지하고 기존 A만큼을 먼저 사용한 뒤 새 B만큼 다시 사용할 수 있습니다 (총 A+B 탈취). 이를 프론트러닝 공격이라 합니다.

안전한 패턴:

  • · allowance를 0으로 먼저 초기화한 후 새 값을 설정 (2단계 approve). 단, 두 트랜잭션 사이에도 프론트러닝이 가능하므로 위험을 줄일 뿐 완전한 해결책은 아닙니다.
  • · EIP-2612 permit(): 오프체인 서명으로 approve를 대체하여 온체인 approve 트랜잭션을 제거함으로써 프론트러닝 위험을 크게 완화합니다. (단, permit 서명 자체도 mempool에서 추출될 수 있으므로 완벽한 방지는 아닙니다.)
  • · 참고: OpenZeppelin v4의 increaseAllowance() / decreaseAllowance()는 v5에서 제거되었습니다. 2단계 approve 또는 permit 방식을 사용하세요.
// 안전한 2단계 approve 예시 await token.approve(spender, 0); // 1. 먼저 0으로 리셋 await token.approve(spender, newAmount); // 2. 새 값 설정

퍼셋(Faucet)이란?

퍼셋은 수도꼭지를 뜻합니다. 정수기에서 물을 받듯이, 테스트용 토큰을 무료로 지급받을 수 있는 스마트 컨트랙트입니다. 단, 남용을 막기 위해 쿨다운 시간이 있어 일정 시간이 지나야 다시 받을 수 있습니다.

TokenFaucet.sol Solidity 
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/IERC20.sol";
import "@openzeppelin/contracts/token/ERC20/utils/SafeERC20.sol";
import "@openzeppelin/contracts/access/Ownable.sol";
import "@openzeppelin/contracts/utils/ReentrancyGuard.sol";

/**
 * @title TokenFaucet
 * @dev 테스트용 토큰을 주기적으로 지급하는 퍼셋 컨트랙트
 *      SafeERC20으로 transfer 반환값 검증, ReentrancyGuard로 재진입 방어
 */
contract TokenFaucet is Ownable, ReentrancyGuard {
  using SafeERC20 for IERC20;
  IERC20 public token;
  uint256 public claimAmount;
  uint256 public cooldownTime;

  // 주소별 마지막 클레임 시각 기록
  mapping(address => uint256) private _lastClaimed;

  event TokensClaimed(
    address indexed claimant,
    uint256 amount,
    uint256 timestamp
  );

  constructor(
    address tokenAddress,
    uint256 _claimAmount,
    uint256 _cooldownTime
  ) Ownable(msg.sender) {
    token = IERC20(tokenAddress);
    claimAmount = _claimAmount;
    cooldownTime = _cooldownTime;
  }

  /**
   * @dev 퍼셋 현재 잔액 조회
   */
  function faucetBalance() external view returns (uint256) {
    return token.balanceOf(address(this));
  }

  /**
   * @dev 다음 클레임 가능 시각 조회
   */
  function nextClaimTime(address account)
    external view returns (uint256)
  {
    return _lastClaimed[account] + cooldownTime;
  }

  /**
   * @dev 토큰 클레임 - 쿨다운 확인 후 지급
   */
  // CEI 패턴: Checks → Effects → Interactions
  function claim() external nonReentrant returns (bool) {
    // 1. Checks (조건 검증)
    require(
      block.timestamp >= _lastClaimed[msg.sender] + cooldownTime,
      "Faucet: cooldown active"
    );
    require(
      token.balanceOf(address(this)) >= claimAmount,
      "Faucet: insufficient balance"
    );

    // 2. Effects (상태 변경을 외부 호출보다 먼저!)
    _lastClaimed[msg.sender] = block.timestamp;

    // 3. Interactions (SafeERC20으로 반환값 자동 검증)
    token.safeTransfer(msg.sender, claimAmount);

    emit TokensClaimed(msg.sender, claimAmount, block.timestamp);
    return true;
  }

  /**
   * @dev 오너만 클레임 수량 변경 가능
   */
  function setClaimAmount(uint256 amount) external onlyOwner {
    claimAmount = amount;
  }

  /**
   * @dev 오너만 쿨다운 시간 변경 가능
   */
  function setCooldownTime(uint256 time) external onlyOwner {
    cooldownTime = time;
  }

  /**
   * @dev 오너가 퍼셋 잔액 회수
   */
  function withdraw() external onlyOwner nonReentrant {
    uint256 balance = token.balanceOf(address(this));
    token.safeTransfer(owner(), balance);
  }
}

보안 체크포인트

  • · SafeERC20: transfer() 반환값을 자동 검증합니다. 비표준 토큰(USDT 등)도 안전하게 처리합니다.
  • · ReentrancyGuard: nonReentrant modifier로 재진입 공격을 방어합니다.
  • · CEI 패턴: Checks → Effects → Interactions 순서로 상태 변경이 외부 호출보다 먼저 실행됩니다.
