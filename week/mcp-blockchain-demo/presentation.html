3. README.md의 핵심 질문

“MCP 서버만 만든 것인가, client/server 구조가 분리되어 있는가?”

답: 이 프로젝트는 서버만 있는 구조가 아니라, MCP client가 server에 연결하고 tool을 발견·호출하는 흐름을 명확히 보여주는 데모입니다.

Server 역할

  • 도메인 기능을 MCP tool로 제공
  • Open-Meteo / CoinGecko 같은 외부 API 호출
  • tool result를 client로 반환

Client 역할

  • stdio로 MCP server 연결
  • list_tools()로 tool discovery
  • call_tool()로 실제 인자와 함께 tool 호출

week/mcp-blockchain-demo/가 README의 중심입니다.

week/mcp-blockchain-demo/
├── README.md                 # 영어 / GitHub 공개용
├── README_KR.md              # 한국어 / 수업 설명용
├── pyproject.toml            # Python 3.11+, uv 의존성
├── run_all_demos.sh          # CLI demo 전체 실행

├── web_app.py                # Starlette backend + MCP client
├── web/index.html            # 브라우저 데모 콘솔
├── servers/
│   ├── weather_server.py     # Weather MCP server
│   └── coin_server.py        # Coin/Blockchain MCP server
├── clients/test_mcp_client.py# CLI MCP client
└── assets/screenshots/       # 발표용 스크린샷

설명 전략

  • 먼저 clients/와 servers/ 분리를 보여줍니다.
  • 그 다음 web_app.py가 웹 backend이면서 MCP client 역할을 한다고 설명합니다.
  • 마지막으로 CLI와 Web UI가 같은 MCP 원리를 공유한다고 연결합니다.

7. CLI 데모 흐름

터미널에서는 3단계만 보여주면 됩니다.

실행 명령

cd week/mcp-blockchain-demo
uv sync
./run_all_demos.sh

# 개별 실행
uv run python clients/test_mcp_client.py weather
uv run python clients/test_mcp_client.py coin

학생에게 보여줄 출력 구조

[1/3] MCP 세션 초기화
[2/3] Tool discovery
[3/3] Tool call 실행

문제 해결 시에는 --verbose로 MCP/HTTP 내부 로그까지 확인할 수 있습니다.

12. 학생 프로젝트 방향

README가 제시하는 기말 프로젝트 요구사항

필수 방향

  • MCP server를 최소 1개 만든다.
  • tool을 최소 2개 이상 노출한다.
  • client에서 list_tools(), call_tool()로 검증한다.
  • 블록체인/코인 관련 API 또는 로컬 데이터셋을 사용한다.

예시 tool

get_coin_pricecompare_coinsget_wallet_balanceget_latest_blocksget_token_metadataanalyze_transaction_hash

14. 결론

발표의 핵심 메시지 3가지

README의 중심

이 저장소에서 가장 중요한 설명은 MCP demo의 client/server 분리입니다.

증명 방식

CLI client와 Web backend가 모두 MCP client로 server tool을 발견하고 호출합니다.

확장 방향

학생들은 블록체인/코인 도메인 tool을 직접 설계해 MCP 최종 프로젝트로 확장할 수 있습니다.

Tip: 발표 중에는 “server만 구현한 것이 아니라, client가 list_tools/call_tool로 검증한다”는 문장을 반복하면 전체 구조가 선명해집니다.
