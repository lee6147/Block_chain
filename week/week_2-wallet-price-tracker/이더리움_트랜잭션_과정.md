# 이더리움 트랜잭션 전송 과정 — 완전 해설

> Blockchain Lab #2 — Sepolia 테스트넷에서 ETH 전송이 어떻게 이루어지는지 단계별로 설명

---

## 전체 흐름 요약

```
사용자 (브라우저)
    │  POST /api/send-eth { to, amount }
    ▼
Node.js 서버 (server.js)
    │  1. 트랜잭션 객체 생성
    │  2. Private Key로 디지털 서명
    ▼
Infura (JSON-RPC 노드)
    │  3. eth_sendRawTransaction 호출
    │  4. 이더리움 P2P 네트워크에 브로드캐스트
    ▼
이더리움 P2P 네트워크
    │  5. 전 세계 노드에 전파
    ▼
멤풀 (Mempool, 대기열)
    │  6. 검증자(Validator)가 트랜잭션 선택
    ▼
검증 & 블록 생성
    │  7. 서명 검증, 잔액 확인, 블록에 포함
    ▼
블록체인에 기록 (불변)
    │  8. receipt 반환
    ▼
서버 → 브라우저에 결과 응답
```

---

## 1단계: 사용자 요청

사용자가 브라우저에서 "ETH 전송" 버튼을 누르면 서버로 HTTP 요청이 전송된다.

```
POST http://localhost:5500/api/send-eth
Content-Type: application/json

{
  "to": "0x24F34Ee5Ef0888157b09281B535eD6612D22b7c6",
  "amount": "0.001"
}
```

- **to**: 받는 사람의 이더리움 지갑 주소 (42자리 16진수)
- **amount**: 보낼 ETH 수량 (문자열)

### 왜 서버를 거치는가?

Private Key(개인키)는 절대 브라우저(클라이언트)에 노출되면 안 된다. 서버에서만 보관하고, 서버가 대신 서명하여 전송한다. 실제 서비스에서는 HSM(Hardware Security Module)이나 KMS(Key Management Service)를 사용한다.

## 2단계: 트랜잭션 객체 생성

서버의 ethers.js가 트랜잭션 객체를 만든다.

```js
var tx = await wallet.sendTransaction({
  to: "0x24F3...7c6",
  value: ethers.parseEther("0.001")  // ETH → Wei 변환
});
```

### 트랜잭션 객체의 구성 요소

| 필드 | 값 (예시) | 설명 |
|------|----------|------|
| `from` | `0x24F3...7c6` | 보내는 사람 주소 (서버 지갑) |
| `to` | `0x24F3...7c6` | 받는 사람 주소 |
| `value` | `1000000000000000` | 전송 금액 (Wei 단위) |
| `nonce` | `0` | 이 지갑의 몇 번째 트랜잭션인지 |
| `gasLimit` | `21000` | 최대 사용 가능 가스량 |
| `gasPrice` | `123811454` | 가스 1단위당 가격 (Wei) |
| `chainId` | `11155111` | Sepolia 테스트넷 체인 ID |

### Wei란?

이더리움의 최소 단위. 소수점 계산 오류를 방지하기 위해 정수로 처리한다.

```
1 ETH       = 1,000,000,000,000,000,000 Wei  (10^18)
1 Gwei      = 1,000,000,000 Wei              (10^9)
0.001 ETH   = 1,000,000,000,000,000 Wei      (10^15)
```

### Nonce란?

이 지갑에서 발생한 트랜잭션의 순번. 0번부터 시작하여 1씩 증가한다.

- **목적**: 같은 트랜잭션이 두 번 실행되는 것(리플레이 공격)을 방지
- **예시**: nonce 0 → 첫 번째 트랜잭션, nonce 1 → 두 번째 트랜잭션
- nonce 3을 보내려면 0, 1, 2가 먼저 처리되어야 한다 (순서 보장)

### Gas란?

이더리움에서 연산을 수행하는 데 필요한 "연료". 트랜잭션 수수료를 결정한다.

```
실제 수수료 = gasUsed × gasPrice
           = 21,000 × 0.123811454 Gwei
           = 2,600.04 Gwei
           ≈ 0.0000026 ETH
```

- **gasLimit**: 최대 이만큼의 가스를 쓸 수 있다 (초과하면 트랜잭션 실패)
- **gasPrice**: 가스 1단위당 얼마를 낼 것인지 (높을수록 빨리 처리됨)
- **gasUsed**: 실제로 사용된 가스량 (단순 ETH 전송은 항상 21,000)

## 3단계: 디지털 서명

트랜잭션 데이터를 Private Key로 서명한다.

```
트랜잭션 데이터 → 해시(Keccak-256) → Private Key로 ECDSA 서명 → (v, r, s) 서명값
```

### 서명의 원리 (ECDSA)

이더리움은 **타원곡선 디지털 서명 알고리즘(ECDSA)**을 사용한다.

```
┌─────────────────────────────────────────────────────┐
│  Private Key (비공개)                                │
│  a690ebcc99a6306e...                                │
│       │                                             │
│       ▼  타원곡선 연산 (secp256k1)                    │
│  Public Key (공개)                                   │
│  04a1b2c3d4e5f6...                                  │
│       │                                             │
│       ▼  Keccak-256 해시 → 마지막 20바이트            │
│  Address (주소)                                      │
│  0x24F34Ee5Ef0888157b09281B535eD6612D22b7c6         │
└─────────────────────────────────────────────────────┘
```

- **Private Key → Public Key**: 수학적으로 가능 (타원곡선 곱셈)
- **Public Key → Private Key**: 수학적으로 불가능 (이산 로그 문제)
- **서명 → Public Key 복원**: 가능 (ecrecover 함수)

### 서명이 증명하는 것

1. **인증(Authentication)**: 이 트랜잭션은 해당 Private Key의 소유자가 만들었다
2. **무결성(Integrity)**: 트랜잭션 데이터가 서명 후 변조되지 않았다
3. **부인 방지(Non-repudiation)**: 보낸 사람이 나중에 "나는 보내지 않았다"고 부인할 수 없다

### 핵심: Private Key는 전송되지 않는다

```
전송되는 것: 트랜잭션 데이터 + 서명(v, r, s)
전송되지 않는 것: Private Key

→ 누구나 서명을 검증할 수 있지만, Private Key를 알아낼 수는 없다
```

## 4단계: Infura를 통해 네트워크 전송

서명된 트랜잭션을 Infura 노드로 전송한다.

```
우리 서버 ──HTTPS──▶ Infura 노드 ──P2P──▶ 이더리움 네트워크
```

### Infura란?

이더리움 풀 노드를 직접 운영하면:
- 디스크 500GB+ 필요
- 동기화에 수일 소요
- 24시간 서버 유지 필요

**Infura**는 이 노드를 클라우드에서 대신 운영해주는 서비스다. 우리는 HTTPS API 호출만 하면 된다.

### JSON-RPC 프로토콜

Infura와의 통신은 JSON-RPC(Remote Procedure Call) 방식이다.

```json
// 요청 (우리 서버 → Infura)
{
  "jsonrpc": "2.0",
  "method": "eth_sendRawTransaction",
  "params": ["0xf86c0085...서명된 트랜잭션 데이터..."],
  "id": 1
}

// 응답 (Infura → 우리 서버)
{
  "jsonrpc": "2.0",
  "result": "0xca110a19...트랜잭션 해시...",
  "id": 1
}
```

### 주요 JSON-RPC 메서드

| 메서드 | 설명 | 사용 예시 |
|--------|------|----------|
| `eth_blockNumber` | 최신 블록 번호 조회 | `/api/block-info` |
| `eth_getBlockByNumber` | 특정 블록 상세 정보 | `/api/block-info` |
| `eth_getTransactionByHash` | 트랜잭션 조회 | `/api/tx` |
| `eth_sendRawTransaction` | 서명된 트랜잭션 전송 | `/api/send-eth` |
| `eth_getBalance` | 잔액 조회 | `/api/wallet-info` |

## 5단계: 멤풀(Mempool)에서 대기

트랜잭션이 네트워크에 도착하면 **멤풀**에 들어간다.

```
┌─────────────────────────────────────────┐
│              멤풀 (Mempool)              │
│                                         │
│  TX #1: 0.5 ETH 전송 (gasPrice: 30 Gwei) │
│  TX #2: NFT 민팅 (gasPrice: 25 Gwei)     │
│  TX #3: 우리 TX (gasPrice: 0.12 Gwei)    │  ← 여기!
│  TX #4: 토큰 스왑 (gasPrice: 20 Gwei)     │
│  ...수천 개의 대기 중인 트랜잭션...         │
│                                         │
└─────────────────────────────────────────┘
```

- 멤풀 = 아직 블록에 포함되지 않은 트랜잭션들의 대기열
- 검증자는 일반적으로 **gasPrice가 높은 트랜잭션을 먼저 선택** (더 많은 수수료 = 더 빠른 처리)
- Sepolia 테스트넷은 트래픽이 적어서 거의 즉시 처리됨

## 6단계: 검증자(Validator)의 검증

검증자가 멤풀에서 트랜잭션을 꺼내 검증한다.

### 검증 항목

| 검증 항목 | 확인 내용 | 실패 시 |
|----------|----------|--------|
| 서명 검증 | ECDSA 서명이 from 주소와 일치하는가? | 트랜잭션 거부 |
| 잔액 확인 | `value + (gasLimit × gasPrice)` ≤ 보유 잔액? | 트랜잭션 거부 |
| Nonce 확인 | 기대하는 다음 nonce와 일치하는가? | 대기 또는 거부 |
| Gas 확인 | gasLimit이 충분한가? | 트랜잭션 실패 (가스 소모) |
| 체인 ID | 올바른 네트워크인가? | 트랜잭션 거부 |

### 검증자란? (PoS — Proof of Stake)

이더리움은 2022년 9월 "The Merge" 이후 **지분증명(PoS)** 방식을 사용한다.

```
이전 (PoW): 채굴자가 수학 문제를 풀어서 블록 생성 (GPU, 전기 소모)
현재 (PoS): 검증자가 32 ETH를 예치하고 블록 생성 권한을 받음
```

- 검증자는 32 ETH를 스테이킹(예치)해야 참여 가능
- 무작위로 블록 생성자(proposer)가 선택됨
- 부정행위 시 스테이킹한 ETH가 삭감(slashing)됨

## 7단계: 블록에 포함

검증을 통과한 트랜잭션이 새 블록에 포함된다.

```
블록 #10414977
┌──────────────────────────────────────┐
│  Block Header                        │
│  ├─ parentHash: 이전 블록 해시        │
│  ├─ timestamp: 1741...               │
│  ├─ miner/coinbase: 검증자 주소       │
│  ├─ gasUsed: 블록 전체 가스 사용량     │
│  └─ stateRoot: 상태 트리 루트 해시     │
│                                      │
│  Transactions                        │
│  ├─ TX 0: ...                        │
│  ├─ TX 1: ...                        │
│  ├─ TX 2: 우리 트랜잭션 (0.001 ETH)   │  ← 여기!
│  ├─ TX 3: ...                        │
│  └─ ... (총 N개)                      │
└──────────────────────────────────────┘
         │
         ▼ 해시로 연결
블록 #10414978
┌──────────────────────────────────────┐
│  parentHash: 블록 #10414977의 해시    │
│  ...                                 │
└──────────────────────────────────────┘
```

### 상태 변경 (State Transition)

트랜잭션이 블록에 포함되면 이더리움의 **전역 상태**가 업데이트된다.

```
보내는 사람 (0x24F3...):
  잔액: 0.050000000000000000 ETH
     → 0.049997399959466000 ETH  (-0.001 ETH - 가스비)
  nonce: 0 → 1

받는 사람 (0x24F3..., 동일 주소):
  잔액: +0.001 ETH
```

이 경우 자기 자신에게 보냈으므로 실질적으로 **가스비만 소모**되었다.

```
실제 소모 = 0.05 - 0.049997399959466 = 0.000002600040534 ETH ≈ 가스비
```

## 8단계: 결과 반환

서버가 트랜잭션 영수증(receipt)을 받아 브라우저에 응답한다.

```js
// ethers.js가 블록 포함을 기다림
var receipt = await tx.wait();
```

### 반환되는 정보

```json
{
  "hash": "0xca110a19dfdeda048d2b6843ee8acb04ff970d99993a83993f329131c26bb286",
  "from": "0x24F34Ee5Ef0888157b09281B535eD6612D22b7c6",
  "to": "0x24F34Ee5Ef0888157b09281B535eD6612D22b7c6",
  "amount": "0.001 ETH",
  "blockNumber": 10414977,
  "status": "성공",
  "sepoliaLink": "https://sepolia.etherscan.io/tx/0xca110a19..."
}
```

### Transaction Receipt의 status

| status 값 | 의미 | 원인 |
|-----------|------|------|
| `1` | 성공 | 정상 처리 |
| `0` | 실패 | 가스 부족, 컨트랙트 에러 등 (가스비는 소모됨!) |

> **주의**: 트랜잭션이 실패해도 가스비는 돌려받지 못한다. 검증자가 연산을 수행했기 때문이다.

## 메인넷 vs 테스트넷 비교

| 항목 | 메인넷 (Mainnet) | Sepolia (Testnet) |
|------|-----------------|-------------------|
| Chain ID | 1 | 11155111 (0xaa36a7) |
| ETH 가치 | 실제 돈 ($2,000+/ETH) | 0원 (무료) |
| ETH 획득 | 거래소 구매 | Faucet 무료 수령 |
| 용도 | 실제 거래, DeFi, NFT | 개발, 테스트, 학습 |
| 합의 방식 | PoS (동일) | PoS (동일) |
| 블록 시간 | ~12초 | ~12초 |
| Etherscan | etherscan.io | sepolia.etherscan.io |

Sepolia는 메인넷과 **동일한 프로토콜**을 사용하므로, 여기서 테스트한 코드는 주소와 네트워크 설정만 바꾸면 메인넷에서도 그대로 동작한다.

```
┌─────────────────────────────────────────────────┐
│                    브라우저                       │
│  HTML + CSS (Tailwind) + JS (ethers.js CDN)     │
│  MetaMask 확장 프로그램 (지갑 연결)               │
└─────────────────────┬───────────────────────────┘
                      │ HTTP (localhost:5500)
┌─────────────────────▼───────────────────────────┐
│               Node.js HTTP 서버                  │
│  server.js (순수 Node.js, Express 없음)          │
│  ├─ 정적 파일 서빙 (html, css, js)               │
│  ├─ CoinGecko API 프록시 (CORS 우회)            │
│  ├─ ethers.js v6 (트랜잭션 서명/전송)             │
│  └─ 30초 캐시 (API 호출 제한 대응)               │
└─────────────────────┬───────────────────────────┘
                      │ HTTPS (JSON-RPC)
┌─────────────────────▼───────────────────────────┐
│                  Infura 노드                     │
│  이더리움 풀 노드를 클라우드에서 운영              │
│  ├─ 메인넷: mainnet.infura.io                    │
│  └─ Sepolia: sepolia.infura.io                   │
└─────────────────────┬───────────────────────────┘
                      │ P2P 프로토콜
┌─────────────────────▼───────────────────────────┐
│            이더리움 블록체인 네트워크              │
│  전 세계 수만 개의 노드가 동일한 상태를 유지       │
│  블록 생성 → 전파 → 합의 → 확정                  │
└─────────────────────────────────────────────────┘
```
